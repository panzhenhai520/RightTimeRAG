# 智能体优化步骤

## 1. 目标

本文件用于在现有 `智能体自定义平台开发改进计划-v2.md` 已完成的基础上，继续优化三类能力：

1. 更完整的可视化连线类型校验。
2. 更丰富、可组合、可测试的智能体节点库。
3. 真实端到端场景回归和压测体系。

本计划的执行原则是：每完成一个开发步骤，必须立即运行对应测试；测试通过后才能进入下一步。每日回归和发版前回归只作为补充保障，不替代开发完成后的即时测试。

## 2. 测试策略说明

### 2.1 开发完成后立即测试

每个步骤完成后必须立即执行对应测试，测试通过才能进入下一步骤。

最低要求：

| 改动类型 | 必跑测试 |
|---|---|
| 后端服务、节点、校验规则 | `tools/run_agent_platform_regression.sh` |
| 前端节点、画布、连线、运行详情 | `cd web && npm run build` |
| 语音/ASR/TTS 节点 | 本地服务健康检查 + 对应 ASR/TTS 路由测试 |
| 端到端流程 | 对应 E2E 场景脚本 |
| 压测工具 | 小并发 smoke + 目标并发压测 |

### 2.2 E2E Regression 为什么不是只“每天”跑

“每天跑”是为了发现非当前提交引入的问题，例如依赖服务状态变化、模型服务异常、缓存污染、队列堆积、跨天数据状态变化。它不是开发阶段的主要验收方式。

本项目采用以下规则：

1. 开发完成后立即跑相关 E2E，这是硬门禁。
2. 每天定时跑 E2E，是补充巡检。
3. 发版前跑完整 E2E，是上线前确认。

### 2.3 “发版前”是什么意思

本项目里的“发版前”指以下任一动作发生之前：

1. 推送到生产使用分支。
2. 重启正式服务给用户使用。
3. 打包并部署前端静态产物。
4. 发布新的智能体模板给普通用户使用。
5. 合并到主分支或打 tag。

发版前回归不是替代开发测试，而是防止多个已通过的小改动组合后出现端到端问题。

## 5. 阶段 2：后端连线类型校验

| 项目 | 内容 |
|---|---|
| 目标 | 发布前由后端阻断错误连线，前端校验不能成为唯一安全线。 |
| 实现内容 | 在 `AgentValidationService` 中加入类型兼容规则；校验边的 source output 和 target input 是否兼容。 |
| 边界 | 不改变现有节点执行逻辑；旧 DSL 缺少端口信息时给 warning，不直接破坏历史流程。 |
| 即时测试 | 增加合法连线、非法连线、旧 DSL 兼容测试；运行 `tools/run_agent_platform_regression.sh`。 |
| 验收条件 | 错误连线发布失败；warning 不阻断旧流程；错误信息能指出源节点、源输出、目标节点、目标输入。 |

默认兼容规则：

| 来源类型 | 目标类型 | 结果 |
|---|---|---|
| `FileAsset` | `FileAsset` | 允许 |
| `FileAsset` | `TextDocument` | 不允许，必须经过 `FileParser` |
| `TextChunk[]` | `String`/LLM context | 允许，由节点显式摘要或拼接 |
| `SQLResult` | `TableData` | 允许 |
| `TableData` | `JSON` | 允许，但建议经过显式转换节点 |
| `Number` | SQL 参数 | 允许 |
| `Artifact` | `Message`/下载输出 | 允许 |
| `Artifact` | LLM prompt | 不允许，必须经过读取/解析节点 |
| `AudioAsset` | ASR 输入 | 允许 |
| `String` | TTS 输入 | 允许 |
| `VoiceReply` | `Message`/voice service | 允许 |

## 6. 阶段 3：前端画布端口与即时连线校验

| 项目 | 内容 |
|---|---|
| 目标 | 用户拖拽连线时就能看到类型是否兼容，减少发布后失败。 |
| 实现内容 | React Flow 节点显示输入/输出端口；连线时读取 manifest；非法连线阻止或标红；变量选择器只显示兼容变量。 |
| 边界 | 不重做整个画布视觉；不改变已有节点位置和旧流程打开方式。 |

UI 要求：

1. 不让画布变得拥挤，端口标签默认简洁显示。
2. 鼠标悬停端口显示类型和说明。
3. 连线失败提示使用明确原因，例如：`FileAsset 不能直接连接到 LLM prompt，请先连接 FileParser`。
4. 发布校验和前端校验使用同一套兼容矩阵，避免前端允许、后端拒绝。

### 10.1 语音节点是否连接服务器上的 TTS 和 ASR

是。语音/多智能体类节点应连接当前服务器上已经部署或计划纳入启动栈的本地语音能力。

默认本地依赖：

| 能力 | 默认服务 | 默认地址 | 运行方式 | 是否需要外网 |
|---|---|---|---|---|
| TTS | CosyVoice3 | `http://127.0.0.1:50001` | Docker 容器 `cosyvoice3-gpu3`，GPU3 | 运行时不需要 |
| ASR 高精度 | Qwen3-ASR-1.7B | `http://127.0.0.1:9900` | `newapp/src/qwen3-asr/start_qwen3_asr.sh` 或 systemd | 运行时不需要 |
| ASR 低延迟/备用 | SenseVoice | `http://127.0.0.1:9997` | Xinference 服务 | 运行时不需要 |
| LLM 推理 | DeepSeek V4 Flash | `http://127.0.0.1:8106` | `ds4-server-ragflow.service` | 运行时不需要 |
| 嵌入/重排 | bge-m3 / bge-reranker-large | `http://127.0.0.1:9997` | Xinference 服务 | 运行时不需要 |
| RAGFlow 后端 | RAGFlow source | `http://127.0.0.1:9388` | `ragflow-source.service` | 运行时不需要 |

说明：

1. 本地模型权重已经在服务器时，运行时不需要连接外网。
2. 首次下载模型、更新镜像、安装依赖时可能需要外网或内网镜像源，但这属于部署阶段，不属于节点运行阶段。
3. 如果用户主动配置云 ASR、云 TTS、SerpApi、外部数据库或外部 Webhook，则对应节点会依赖外网或专线网络。
4. 节点必须把依赖写入 manifest，例如 `requires_service: ["cosyvoice3", "qwen3_asr"]`。
5. 如果依赖服务未启用，节点应在发布校验或运行前健康检查时给出明确错误，而不是运行到一半才失败。

### 11.1 最终一次集成测试：真人英语老师教学评测智能体

最终集成测试统一收敛为一次完整智能体创建和运行测试：使用新增节点建立一个模拟真人老师思维的英语教学智能体。

这个测试算最终集成测试，但必须显式覆盖所有新增节点类别；如果只输入文字和学生语音，不上传教学材料、不接外部评分、不写数据库、不生成图表报告，则只能算语音教学链路测试，不能算最终集成测试。

集成测试智能体名称：

```text
真人英语老师教学评测智能体
```

测试输入：

| 输入 | 类型 | 说明 |
|---|---|---|
| 教学目标文本 | `String` | 例如：教会学生念一段英文 |
| 英文朗读文本 | `String` | 老师要教学生朗读的英文 |
| 学生语音 | `AudioAsset` | 学生跟读录音 |
| 教学材料 | `FileAsset[]` | PDF/DOCX/图片，用于覆盖文件解析、OCR、检索、引用 |
| 学生名单/历史成绩 | `FileAsset` | Excel fixture，用于覆盖表格节点 |
| 外部裁判评分 | `JSON` | Webhook 或人工补录 |
| 评分权重 | `JSON` | 例如：自评 0.6，外部评分 0.4 |

流程必须覆盖：

1. 文本输入和文件上传。
2. 教学材料解析、切片、OCR、检索和引用。
3. 学生名单/历史成绩 Excel 读取。
4. LLM 生成教学方案、教学步骤和评分规则。
5. TTS 生成老师示范朗读音频。
6. ASR 将学生语音识别成文字。
7. 根据识别文字计算准确率和多维评分。
8. 接收外部裁判评分或人工补录评分。
9. 按权重公式合并综合评分。
10. 使用受控数据库节点动态建立教学活动表和学生成绩表。
11. 写入教学活动日志和学生成绩。
12. 查询历史成绩并生成图表。
13. 生成带图表历史分析的报告。
14. 输出报告下载、音频输出、综合评分和教学反馈。

数据库写入边界：

1. 现有 `ExeSQL` 仍保持只读默认策略。
2. 动态建表和写入必须通过受控节点，例如 `SafeTableEnsure`、`SafeRecordInsert`、`SafeRecordQuery`。
3. 动态表名必须限制在租户和智能体命名空间内，例如 `agent_{agent_id}_teaching_activity`。
4. 不允许任意 DDL/DML，不允许写系统表，不允许跨租户访问。

建议总集成测试命令：

```bash
tools/run_agent_e2e_teacher_english.sh
```

验收条件：

1. 能自动创建并运行 `真人英语老师教学评测智能体`。
2. 老师教学方案、教学步骤、TTS 音频、ASR 文字、评分规则、外部评分、综合评分都完整产生。
3. 教学活动日志和学生成绩写入受控数据库表。
4. 历史成绩图表和分析报告生成并可下载。
5. run trace 能定位每个关键节点。
6. 不影响聊天助手正文格式、retrieving/thinking/answer 样式和普通问答流式节奏。

## 12. 阶段 9：压测体系

| 项目 | 内容 |
|---|---|
| 目标 | 验证并发、大文件、长任务、队列、worker、输出物在真实负载下稳定。 |
| 实现内容 | 引入 `Locust` 或 `k6`；增加 fake LLM；生成测试 PDF、Excel、音频 fixture；输出压测报告。 |

| 验收条件 | 成功率、p95、队列等待、worker 耗时、artifact 成功率、错误定位率达到目标。 |

建议指标：

| 指标 | 目标 |
|---|---|
| 成功率 | >= 99% |
| 任务最终状态一致性 | 100% |
| artifact 生成成功率 | >= 99% |
| 失败任务可追踪率 | 100% |
| 危险 SQL 拦截率 | 100% |

## 16. 不应越界的事项

1. 不为了前端连线校验破坏旧 DSL。
2. 不让 LLM 执行确定性计算，Excel/SQL/聚合必须由代码完成。
3. 不默认依赖外网服务；本地 ASR/TTS/LLM/Embedding 应作为默认链路。
4. 不在事件表写入大文本、大表格、大音频，必须走 artifact 或摘要。
5. 不把前端校验当作安全边界，后端发布校验和运行前校验必须存在。
6. 不让压测直接冲击生产用户正在使用的服务；压测需要隔离数据和明确开关。
7. 不改聊天助手现有回答正文格式。
8. 不改聊天助手的 retrieving / thinking / answer 输出样式。
9. 不改普通问答的流式输出节奏。
10. 不把智能体 run 事件混到聊天助手会话事件里。
11. 不复用前端同一个临时状态对象，避免历史会话污染。

## 16.1 聊天助手与智能体运行时的隔离边界

智能体长任务优化可以使用 `run_id`、事件持久化、SSE 或轮询恢复，但这些能力必须限定在智能体运行时内，不得顺手改动聊天助手的展示链路。

明确边界：

1. 聊天助手回答正文的 Markdown、引用、总结卡片、证据区展示格式保持不变。
2. 聊天助手 `retrieving`、`thinking`、`answer` 三段现有输出样式保持不变。
3. 普通问答的流式输出节奏保持不变，不因为智能体事件重放而变慢或改变分段。
4. 智能体 run event 使用独立命名空间、独立 store、独立恢复逻辑，不能混入聊天助手会话事件。
5. 前端不能复用同一个临时状态对象同时承载聊天助手和智能体运行状态。
6. 如果未来要迁移聊天助手到 `run_id + event replay` 模式，必须单独立项、单独测试、单独回退开关，不能作为本优化计划的附带改动。
