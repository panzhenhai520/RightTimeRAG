# 多智能体测试任务

生成日期：2026-07-01

## 一、目的

本文档从 `智能体自定义平台开发改进计划-v3.md` 中拆分测试、回归、smoke、端到端验收、压测和安全验证任务。

开发计划回答“要开发什么能力”，本文档回答“如何证明这些能力可用、稳定、安全、可追溯”。

## 二、当前基线

已执行过的平台回归命令：

```bash
tools/run_agent_platform_regression.sh
```

当前记录：

```text
156 passed
```

该记录只能说明当前已有平台底座测试通过，不代表 v3 中新增的四教师业务闭环、文件工作区访问、文档比对、条款冲突判断等能力已经完成。

## 三、测试分层

测试按以下层级组织：

- 单元测试：服务函数、节点函数、schema 校验、权限判断、diff 算法。
- 组件测试：Agent 节点输入输出、错误码、结构化结果。
- 集成测试：API + Service + Node 串联。
- 真实服务 smoke：使用真实 RagFlow 服务、Redis、worker、存储、知识库。
- 端到端测试：从 UI 或外部 API 发起完整智能体流程。
- 压测和恢复测试：并发、超时、取消、worker 重启。
- 安全测试：越权、路径穿越、prompt injection、敏感信息泄露。

### T13：条款匹配和冲突判断测试

覆盖范围：

- ClauseMatcher。
- SemanticComparer。
- ConflictDetector。
- LegalContractConflictDetector。

测试点：

- 相同条款识别为 `same`。
- 同义改写识别为 `equivalent`。
- A 包含 B 识别为 `a_contains_b`。

- 法律要求必须包含争议解决条款，合同缺失，识别为 `missing_in_b`。
- 无法判断时返回 `ambiguous`，不能强行下结论。
- 每条结果有 reason、severity、confidence、evidence。

### T17：工作区写入和 Patch Apply 回归测试

当启用 `WorkspaceFileWriteService` 和 `WorkspacePatchService` 后执行。命令执行能力仍保持独立评估，不包含在本测试范围。

测试点：

- `WorkspaceFileWrite` 支持 create / overwrite / append。
- 写入和 patch 默认需要 dry-run 或人工审批。
- 非白名单目录不能写。
- structured patch 和 unified diff 都能 dry-run。
- patch 上下文冲突时不能半写入。
- expected_hash 不匹配时拒绝写入。
- patch apply 后能 rollback。
- 每次 dry-run、apply、rollback 有 diff、approval 或 audit。
