# 智能体自定义平台开发改进计划

## 1. 背景与目标

当前平台已经具备 Agent Canvas、Dataflow、节点拖拽、发布、单节点调试、SSE 流式输出、文件上传、知识库解析队列等基础能力。但这些能力仍然分散：

- 普通智能体运行主要绑定在一次 HTTP/SSE 请求内，页面切换、刷新、断线后不具备可靠恢复能力。
- Dataflow 已有 Redis 队列和 `task_executor`，但主要服务于文档解析，不是通用 Agent 长任务运行时。
- 文件上传后经常被解析成文本传入 LLM，原始文件资产、解析文本、结构化表格、输出附件没有统一类型和生命周期。
- 后端有部分节点能力，例如 `ExcelProcessor`，但前端表单、节点菜单、输出物展示、下载链路没有完整打通。
- 节点输入输出契约不严格，拖拽编排时无法可靠判断节点之间是否兼容。

本计划目标是将现有平台逐步改造成一个成熟的可视化智能体自定义平台，支持：

1. 稳定的可拖拽业务流程编排。
2. 文件、表格、SQL、LLM、文档导出等业务节点可组合。
3. 长任务异步执行、页面切换不丢状态、断线可恢复。
4. 节点级调试、变量快照、运行 trace、输出物管理。
5. 发布前校验、权限控制、安全边界和渐进式兼容。

## 2. 总体原则

| 原则 | 说明 |
|---|---|
| 基础优先 | 先补类型系统、运行时、文件资产、输出物，再补业务节点。 |
| 渐进兼容 | 保留现有 DSL、现有 Agent 会话和 Dataflow 能力，新增字段默认兼容旧流程。 |
| 运行可恢复 | 长任务必须有 `run_id`、事件持久化、状态查询和取消能力。 |
| 节点契约化 | 所有节点必须声明输入、输出、类型、是否必填、是否支持流式、是否产生附件。 |
| 安全默认 | SQL 默认只读，文件访问必须基于当前用户/租户权限，密钥不能明文暴露给前端。 |
| 可测试 | 每个阶段必须有单元测试、集成测试和 UI 验收场景。 |

## 3. 目标架构

目标架构分为七层：

1. **DSL 层**：保存节点、边、节点参数、变量定义、类型契约、发布版本。
2. **类型系统层**：定义 `FileAsset`、`TextDocument`、`TableData`、`Number`、`SQLResult`、`Artifact` 等标准类型。
3. **运行时层**：使用 `agent_run` 管理一次智能体运行，使用事件流记录运行过程。
4. **任务队列层**：使用 Redis Stream 或现有队列能力，由 `agent_executor` 异步执行长任务。
5. **节点执行层**：每个节点实现统一生命周期和进度上报。
6. **输出物层**：统一管理 docx、xlsx、pdf、csv、图片、json 等下载文件。
7. **前端体验层**：支持拖拽、调试、运行状态、断线恢复、输出物预览和发布校验。

## 4. 分阶段开发计划

### 阶段 0：现状基线与回归保护

| 项目 | 内容 |
|---|---|
| 目标 | 固化当前可用能力，避免后续重构破坏现有聊天、发布、Dataflow、知识库解析。 |

| 规范 | 不改变现有 API 返回结构；新增字段必须可选；旧 DSL 必须能正常加载。 |
| 注意事项 | 当前工作区可能有历史改动，不做大面积重构；先建立测试基线。 |
| 边界 | 本阶段不新增业务功能，只做现状确认和测试保护。 |
| 验收标准 | 新建智能体、发布智能体、运行普通问答、运行法律助手、上传文件、生成 docx、Dataflow 解析均能通过 smoke test。 |

建议测试：

- 前端能进入智能体列表、创建智能体、编辑流程、保存发布。
- 普通 Agent 能流式输出。
- `DocGenerator` 能生成 docx 并下载。
- Dataflow 能完成 `Parser -> Tokenizer -> Extractor`。
- 当前法律助手流程不报错，保留原行为。

| 项目 | 内容 |
|---|---|
| 目标 | 解决上传文件只变成文本，Excel/代码/文档定位节点拿不到原始文件的问题。 |
| 实现内容 | 修改 `Begin`、`UserFillUp`、全局 `sys.files` 的输出结构；同时输出原始文件资产和解析文本。 |

```json
  "sys.file_texts": [

    }
  ]
}
```

|---|---|

### 阶段 4：Agent 长任务异步运行时

| 项目 | 内容 |
|---|---|
| 目标 | 普通智能体不再依赖一次 HTTP/SSE 请求完成，支持后台执行、页面切换、断线恢复。 |
| 实现内容 | 新增 `agent_run`、`agent_run_event`、`agent_run_artifact`；新增 `agent_executor`；新增 run 创建、状态查询、事件拉取、取消 API。 |
| 规范 | 每次运行必须有稳定 `run_id`；事件必须有递增 `seq`；最终结果必须写回会话。 |
| 注意事项 | 先作为新执行模式灰度开启，不直接替换旧 SSE；避免再次污染聊天 UI 状态。 |
| 边界 | 本阶段不要求所有节点都有精细进度，但必须有 queued/running/succeeded/failed/canceled 状态。 |
| 验收标准 | 发起一个长 Agent 任务后，切换到其他历史会话再切回，仍能看到最新进度和最终回答；刷新页面后可恢复；取消按钮有效。 |

建议新增表：

```sql
agent_run(
  id,
  canvas_id,
  session_id,
  message_id,
  user_id,
  tenant_id,
  status,
  progress,
  current_node_id,
  input_payload,
  dsl_snapshot,
  error,
  created_at,
  updated_at,
  finished_at
)

agent_run_event(
  id,
  run_id,
  seq,
  event_type,
  node_id,
  data,
  created_at
)

agent_run_artifact(
  id,
  run_id,
  node_id,
  artifact_id,
  created_at
)
```

建议 API：

| API | 用途 |
|---|---|
| `POST /api/v1/agents/{agent_id}/runs` | 创建运行，立即返回 `run_id`、`session_id`、`message_id` |
| `GET /api/v1/agents/runs/{run_id}` | 查询运行状态 |
| `GET /api/v1/agents/runs/{run_id}/events?since=0` | 拉取事件，可断点续拉 |
| `GET /api/v1/agents/runs/{run_id}/artifacts` | 获取输出物 |
| `POST /api/v1/agents/runs/{run_id}/cancel` | 取消运行 |
| `POST /api/v1/agents/runs/{run_id}/inputs` | 等待用户补充输入时提交输入 |

| 项目 | 内容 |
|---|---|
| 目标 | 所有节点运行过程可追踪、可取消、可恢复。 |
| 实现内容 | 定义节点生命周期事件：`node_started`、`node_progress`、`node_output`、`node_finished`、`node_failed`；改造 Canvas 运行器写入事件。 |

```json

  "node_id": "Agent:LegalReport",

    "progress": 0.45,
    "message": "Generating report..."
  }

```

| 项目 | 内容 |
|---|---|

### 阶段 7：Excel 业务能力补齐

| 项目 | 内容 |
|---|---|
| 目标 | 支持用户手动拖拽创建 Excel 读取、汇总、计算、导出流程。 |
| 实现内容 | 开放 `ExcelProcessor` 前端表单；新增或增强 Excel 字段识别、聚合计算、导出节点；接入 `FileAsset` 和 `Artifact`。 |
| 规范 | Excel 节点输入必须是 `FileAsset` 或 `FileAsset[]`；输出 `TableData`、`summary`、`Artifact`。 |
| 注意事项 | 不要让 LLM 猜列名做计算；列名识别可以用规则 + LLM 辅助，但最终计算必须由代码完成。 |
| 边界 | 第一版支持 xlsx/csv，复杂公式、宏、透视表不纳入。 |
| 验收标准 | 上传 Excel，读取列名，识别“合计/总计/金额合计”列，求和，乘以用户输入系数，输出 B，并可导出 xlsx。 |

建议拆成节点：

1. `ExcelRead`：读取文件，输出 `TableData`。
2. `TableColumnDetect`：按规则/LLM 识别目标列。
3. `TableAggregate`：sum/count/avg/min/max。
4. `NumberCalculate`：乘以系数、公式计算。
5. `ExcelExport`：将 `TableData` 输出为 xlsx Artifact。

| 项目 | 内容 |
|---|---|

## 5. 长任务异步执行专项方案

### 5.1 当前问题

普通 Agent 当前主要在请求内执行：

1. 前端发起 SSE。
2. 后端 `Canvas.run()` 执行节点。
3. 执行过程通过 SSE 返回。
4. 请求结束后才完整持久化会话。

这导致：

- 用户切换页面或刷新后，前端状态容易丢失。
- SSE 断开后，任务是否继续执行不稳定。
- 历史会话和当前运行状态容易互相污染。
- 运行进度无法统一查看。
- 无法可靠取消、恢复、重放。

### 5.2 目标行为

| 场景 | 期望行为 |
|---|---|
| 发起长任务 | 立即创建 `run_id`，会话列表显示运行中。 |
| 切换到其他会话 | 后台继续执行，当前页面显示其他会话内容。 |
| 切回运行会话 | 根据 `run_id` 拉取事件，恢复进度和已输出内容。 |
| 刷新浏览器 | 根据 session/run 状态恢复。 |
| 任务完成 | 写入最终回答、trace、artifact，运行中图标消失。 |
| 任务失败 | 显示失败节点、错误信息，可重试。 |
| 用户取消 | 设置 cancel 标记，worker 停止，状态变 canceled。 |

### 5.3 需要新增的组件

| 组件 | 作用 |
|---|---|
| `AgentRunService` | 创建、更新、查询智能体运行记录。 |
| `AgentRunEventService` | 追加和查询运行事件。 |
| `AgentArtifactService` | 管理运行输出物。 |

### 5.4 开发步骤

| 步骤 | 目标 | 实现内容 | 验收 |
|---|---|---|---|
| 1 | 建表 | 新增 `agent_run`、`agent_run_event`、`agent_run_artifact` migration | 表创建成功，旧数据不受影响 |
| 2 | 创建 run API | `POST /agents/{id}/runs` 返回 `run_id` | 前端可拿到 run_id，不阻塞等待结果 |
| 3 | 队列入队 | 将 run payload 写入 Redis 队列 | Redis 可看到待执行任务 |
| 4 | 后台 worker | 新增 `agent_executor` 消费队列 | worker 能执行一个简单 Agent |
| 5 | 事件持久化 | Canvas 事件写入 `agent_run_event` | 可通过 API 查到 node_started/message/node_finished |
| 6 | 状态更新 | queued/running/succeeded/failed/canceled | 任务状态准确变化 |
| 7 | 前端恢复 | 前端根据 run_id 拉取事件重放 | 切换会话后再切回可恢复 |
| 8 | 取消 | cancel API + 节点检查取消 | 运行中任务可停止 |
| 9 | 输出物 | artifact 绑定 run | 完成后输出物可下载 |
| 10 | 灰度开关 | 新旧运行模式可切换 | 出问题可退回旧 SSE |

### 5.5 注意事项

- 不要直接把旧 SSE 删除。先保留旧模式，新模式用配置开关启用。
- LLM 流式 token 不要逐 token 写数据库，可以按时间或字符批量写。
- 大表格、大文本不要进入 event 表，event 只存摘要和 artifact 引用。
- `run_id` 是唯一运行标识，trace、日志、artifact、取消都应围绕它组织。
- 会话最终消息必须由 worker 在任务完成时写入，不能依赖前端连接是否存在。
- 运行中的会话如果被删除，worker 应检测并取消或标记 orphan。

## 6. 端到端业务样例验收

### 6.1 法律条文解析助手

流程：

1. Begin 上传 PDF。
2. FileParser 解析全文。
3. LawSectionLocator 定位相关编/章/条。
4. Retrieval/Filter 返回命中条文。
5. LLM 生成报告。
6. DocGenerator 输出 docx。
7. Message 展示下载。

验收：

- 上传完整《民法典》，询问婚姻相关条款，能定位第 1041 条以后正文。
- 输出 docx 不再声称“只有目录”。
- 文档下载入口清晰。
- 长任务期间切换会话不丢进度。

### 6.2 Excel + SQL 数据分析助手

流程：

1. Begin 上传 Excel，同时输入系数。
2. ExcelRead 输出列名和表格。
3. TableColumnDetect 找到合计列。
4. TableAggregate 求和。
5. NumberCalculate 乘以系数得到 B。
6. ExeSQL 查询 `A > B`。
7. ExcelExport 导出查询结果。
8. LLM 解读结果。

验收：

- Excel 文件能读取到真实列名。
- B 的计算由代码完成，不由 LLM 猜测。
- SQL 只读且可使用 B 参数。
- 查询结果可下载 xlsx。
- LLM 能基于查询结果生成解释。

## 10. 风险与回滚

| 风险 | 应对 |
|---|---|
| 新运行时影响现有聊天 | 使用开关灰度，新旧 SSE 并存。 |
| DSL 不兼容旧智能体 | 新字段可选，加载时自动补默认值。 |
| 事件量过大 | token 批量写入，长内容转 Artifact。 |
| 文件权限复杂 | Artifact 和 FileAsset 统一走租户/用户校验。 |
| SQL 安全不足 | AST 校验 + 只读账号 + 超时 + 最大行数。 |
| Worker 抢占资源 | Agent executor 独立队列，限制并发。 |
| 前端状态再次污染 | 前端以 `run_id/session_id` 隔离状态，历史消息只读展示。 |

## 11. 最小可落地版本

如果先做一个最小可用版本，建议只做以下内容：

1. 新增 `agent_run` 和 `agent_run_event`。
2. 新增 `POST /agents/{id}/runs`、`GET /agents/runs/{run_id}`、`GET /agents/runs/{run_id}/events`、`POST /agents/runs/{run_id}/cancel`。
3. 新增 `agent_executor`，后台执行普通 Agent Canvas。
4. Canvas 事件写入 run event。
5. 前端会话列表显示运行中状态，切换回来按事件恢复。
6. 运行完成后写入会话消息。
7. 保留旧 SSE 作为回滚路径。

这个最小版本完成后，再继续补文件资产、输出物、Excel、SQL 和法律长文档能力。
