# 智能体自定义平台开发改进计划 v2

## 1. 为什么上一版出现多个“部分完成”

上一版计划按能力层分成 12 个阶段，但实际开发过程中穿插了法律助手、Excel、SQL、下载、长任务体验、前端会话状态等多个端到端问题。为了让具体场景先跑通，采用了“纵向切片”的实现方式：每个切片会同时触碰文件资产、节点、输出物、运行状态和前端展示。因此很多阶段都有第一版能力，但没有完整达到该阶段的验收条件。

从 v2 开始，改为严格阶段门禁：

- 每个阶段必须有明确完成定义。
- 每个阶段必须有自动化测试或可复现 smoke test。
- 未通过测试不进入下一阶段。
- 新需求只能插入到依赖正确的位置，不能打乱基础阶段。
- 多 AI 智能体并发调度纳入后置阶段，依赖稳定长任务运行时、事件恢复、记忆命名空间和结构化输出。

## 3. 阶段门禁规则

每个阶段完成时必须满足以下条件：

1. 后端单元测试通过。
2. 涉及前端时，相关 TypeScript 检查或 `web` build 通过。
3. 涉及 API 时，至少有服务层或路由级测试。
4. 涉及 UI 状态时，必须说明可复现的手工验收路径。
5. 不允许为了进入下一阶段留下已知破坏性问题。

阶段状态只允许三种：

| 状态 | 含义 |
|---|---|
| 未开始 | 没有进入开发 |
| 进行中 | 正在补齐本阶段，不能开始依赖它的后续阶段 |
| 已完成 | 完成定义全部满足，可进入下一阶段 |

## 4. v2 阶段计划

### v2-P0：基线冻结与回归门禁

| 项目 | 内容 |
|---|---|
| 目标 | 把当前已实现的智能体改造能力固化成固定回归测试，避免后续继续改乱。 |
| 实现 | 增加统一回归脚本；纳入 AgentRun、契约、FileParser、ExcelProcessor、SQL Guard、发布校验、下载输出物测试。 |
| 边界 | 不新增业务能力，只做门禁。 |
| 验收 | 一条命令可跑完当前智能体平台核心改造测试，全部通过。 |

完成后才能进入 v2-P1。

### v2-P1：节点契约完整化

| 项目 | 内容 |
|---|---|
| 目标 | 让节点输入输出契约成为后续流程编排、发布校验、多智能体调度的基础。 |
| 实现 | 补齐标准类型：`TextDocument`、`TextChunk`、`TableData`、`SQLResult`、`Artifact[]`；统一 `input_schema/output_schema/runtime_capabilities`；补后端测试。 |
| 边界 | 本阶段不强制改造所有前端连线交互，只保证后端契约稳定可读。 |
| 验收 | 契约测试通过；核心节点能输出标准 schema；旧 DSL 可加载。 |

### v2-P2：文件资产、解析文本、切片分离

| 项目 | 内容 |
|---|---|
| 目标 | 上传文件同时服务 LLM、Excel、FileParser、DocGenerator，不再混成单一文本。 |
| 实现 | 完整输出 `sys.files`、`sys.file_assets`、`sys.file_texts`、`sys.file_chunks`；Begin/UserFillUp 保持兼容。 |
| 边界 | 不做正式知识库索引，只做运行级文件结构。 |
| 验收 | Excel 能拿原始文件；LLM 能拿解析文本；FileParser 能拿文件资产并切片。 |

### v2-P3：输出物 Artifact 完整化

| 项目 | 内容 |
|---|---|
| 目标 | 统一 docx/xlsx/csv/pdf/json 输出物的生成、下载、权限和 run 绑定。 |
| 实现 | 补 artifact 元数据结构；绑定 `run_id/node_id`；统一下载信息；保留旧 `DocGenerator@download` 兼容。 |
| 边界 | 第一版可继续使用现有 storage，不强制新建物理存储。 |
| 验收 | 法律助手、Excel 导出、DocGenerator 下载全部走统一结构。 |

### v2-P4：可靠长任务运行时

| 项目 | 内容 |
|---|---|
| 目标 | 普通智能体运行不再依赖单个 HTTP/SSE 连接。 |
| 实现 | 用 Redis Stream 或现有可靠队列承载 `agent_run_queue`；新增独立 `agent_executor`；run 状态、事件、artifact 持久化；支持取消和重试。 |
| 边界 | 保留旧 SSE 模式作为回滚。 |
| 验收 | 发起任务后切换会话、刷新页面、SSE 断开、短暂后端重启后可恢复或明确失败，不丢最终状态。 |

### v2-P5：节点生命周期、进度和取消

| 项目 | 内容 |
|---|---|
| 目标 | 每个节点可追踪、可取消、可定位失败原因。 |
| 实现 | 统一事件：`workflow_started`、`node_started`、`node_progress`、`node_output`、`node_finished`、`node_failed`、`workflow_finished`、`workflow_failed`；LLM token 批量写事件。 |
| 边界 | 不要求所有第三方工具精确百分比，但必须有状态。 |
| 验收 | trace 能展示当前节点、耗时、输入摘要、输出摘要、错误信息。 |

### v2-P6：前端长任务恢复体验

| 项目 | 内容 |
|---|---|
| 目标 | 会话切换、刷新、历史会话查看时不污染状态。 |
| 实现 | 前端只以 `run_id/session_id` 拉取后端事件；历史消息静态展示；运行中会话在列表显示状态；切回后重放事件。 |
| 边界 | 不重做整体视觉，只解决状态正确性。 |
| 验收 | 运行中切到历史会话，历史不会出现活动动画；切回运行会话能看到进度或最终回答。 |

### v2-P7：Excel 业务组件化

| 项目 | 内容 |
|---|---|
| 目标 | 支持用户拖拽生成 Excel 读取、聚合、计算、导出流程。 |
| 实现 | 在现有 `ExcelProcessor` 基础上拆出或显式配置 read/aggregate/calculate/export；输出 `TableData/Number/Artifact`。 |
| 边界 | 第一版支持 xlsx/csv；不支持宏、复杂公式、透视表。 |
| 验收 | 上传 Excel，识别合计列，求和，乘以系数 B，导出 xlsx，LLM 解读。 |

### v2-P8：SQL 安全与变量化查询

| 项目 | 内容 |
|---|---|
| 目标 | 安全使用上游变量执行只读 SQL。 |
| 实现 | 在现有 SQL Guard 基础上补强 AST 校验、只读账号建议、超时、最大行数、结构化 `SQLResult` 到 Excel 导出。 |
| 边界 | 不支持写库。 |
| 验收 | `{ExcelProcessor@result}` 可安全进入 SQL；危险 SQL 被拦截；结果可导出并交给 LLM。 |

### v2-P9：长文档与 Dataflow 桥接

| 项目 | 内容 |
|---|---|
| 目标 | 法律条文、合同、长 PDF 不再全文塞入提示词。 |
| 实现 | FileParser 支持运行级解析、切片、页码、条号、临时检索；必要时接入临时向量索引。 |
| 边界 | 不污染正式知识库。 |
| 验收 | 上传完整民法典，询问婚姻条款，能定位正文条款并输出引用。 |

### v2-P10A：后端调试观测结构

| 项目 | 内容 |
|---|---|
| 目标 | 后端 trace 能稳定表达一次运行的时间线、耗时、节点状态、错误和输出物。 |
| 实现 | `GET /agents/runs/{run_id}/trace` 返回 `timeline`、`duration`、节点输入输出摘要、错误、下载物。 |
| 边界 | 大字段摘要展示，完整内容下载；敏感信息脱敏。 |
| 验收 | trace 单元测试通过，能通过 API 定位失败节点和运行耗时。 |

### v2-P10B：前端运行详情页

| 项目 | 内容 |
|---|---|
| 目标 | 用户能在界面中查看一次 Agent run 的运行详情。 |
| 实现 | 前端运行详情页/抽屉展示 timeline、节点状态、耗时、错误、输出物下载、输入输出摘要。 |
| 边界 | 不展示大字段全文；敏感信息以后端脱敏为准。 |
| 验收 | 打开任意 run 可定位失败节点；前端 build 通过。 |

### v2-P11：发布前校验与版本治理

| 项目 | 内容 |
|---|---|
| 目标 | 发布前发现流程错误。 |
| 实现 | 类型不兼容、变量不存在、文件未处理、危险 SQL、无输出节点、缺模型等校验；error 阻止发布，warning 允许发布。 |
| 边界 | 对旧流程渐进提示，不直接破坏。 |
| 验收 | 错误流程无法发布，warning 可发布但明确提示。 |

### v2-P12：模板和组件库产品化

| 项目 | 内容 |
|---|---|
| 目标 | 用户能从模板快速创建法律、Excel+SQL、PDF 报告、合同审查智能体。 |
| 实现 | 模板统一使用新类型、新 Artifact、新运行时；参数可编辑；无硬编码业务文件。 |
| 边界 | 第一批覆盖高频场景。 |
| 验收 | 从模板创建后可端到端运行、下载、调试。 |

### v2-P13：会议记忆与多 AI 智能体并发调度

| 项目 | 内容 |
|---|---|
| 目标 | 支持同一会议上下文下，多个 AI 智能体注入不同记忆并发生成不同回答，供 voice service 保存和播报。 |
| 依赖 | 必须在 v2-P4、P5、P6 稳定后开始；否则并发运行会放大状态串线和恢复问题。 |
| 实现 | 新增会议调度 Orchestrator；定义 `meeting_id/turn_id/agent_id` 记忆命名空间；同一 `meeting_context` fan-out 到多个 agent run；收集结构化输出；写回 shared memory 和 agent memory。 |
| 输出 | 每个智能体返回 `reply_text`、`memory_delta`、`citations`、`voice_params`、`run_id`。 |
| 边界 | RAGFlow 负责思考和生成；voice service 负责 ASR、保存、TTS、播放、回调。 |
| 验收 | 一轮会议输入可并发调用 4 个智能体；4 个 run 互不串记忆；同一上下文下输出不同角色话术；voice service 可按结果播放。 |

7. v2-P6：前端长任务恢复体验。
8. v2-P7：Excel 业务组件化。
9. v2-P8：SQL 安全与变量化查询。
10. v2-P9：长文档与 Dataflow 桥接。
11. v2-P10A：后端调试观测结构。

## 6. 当前起点

当前已有能力作为 v2 的基础输入：

- `AgentRunService`：Redis 运行状态、事件、进度、下载摘要。
- 背景 run API：已支持轻量后台运行，但还不是可靠队列 worker。
- `FileParser`：已有解析、切片、关键词、条号范围匹配。
- `ExcelProcessor`：已有读取、聚合、导出能力。
- `SQL Guard`：已有只读校验和变量安全替换第一版。
- `ArtifactService`：已有 storage 下载信息封装。
- `AgentValidationService`：已有发布校验第一版。
- 法律和 Excel SQL 模板：已有第一版。

因此，下一步从 v2-P0 开始补门禁，然后进入 v2-P1，而不是直接做多智能体调度。
