# 智能体自定义平台开发改进计划 v4

生成日期：2026-07-01

## 一、本版目标

v3 解决的是“智能体能不能安全读取文件、规范化文档、抽取条款、比对差异、生成报告”。

v4 要解决的是更上层的问题：让 RagFlow 智能体具备类似 Codex 的任务理解、任务分解、前置条件判断、递归执行、结果核对、失败恢复、回到上一级任务继续推进的能力。

目标不是让一个 LLM 节点一次性回答，而是建设一个可审计、可恢复、可测试的任务执行闭环：

```text
用户目标
  -> 意图识别
  -> 目标规范化
  -> 上下文收集
  -> 层级任务分解
  -> 原子任务定义
  -> 前置条件检查
  -> 子任务执行
  -> 结果核对
  -> 返回父任务
  -> 必要时重新规划
  -> 输出最终结果和审计报告
```

## 二、v4 要补齐的核心能力

### 1. 目标识别能力

智能体需要判断用户真正要完成什么，而不是只识别关键词。

例如用户说：

```text
把最近上次写的某文档找出来，我需要调整，新增一些内容。
```

系统应识别为：

- 用户目标：修改已有文档。
- 隐含任务：找到目标文档。
- 隐含任务：读取文档结构。
- 隐含任务：理解新增内容。
- 隐含任务：判断新增内容和原结构是否同级。
- 隐含任务：提出结构调整建议。
- 隐含任务：在用户允许或策略允许时修改文档。
- 验收标准：文档已修改，新增内容放在合适层级，修改有 diff 和证据。

### 2. 任务路径理解能力

系统不仅要知道目标，还要知道完成目标需要经过哪些路径。

例如“修改文档”可能有多条路径：

- 文件路径已知：直接读取目标文件。
- 文件路径未知：先按文件名、时间、内容相似度找文件。
- 用户新增内容结构不清晰：先分类和重组，再修改。
- 文档已有结构不适配：先建议新层级，再合并。
- 修改风险较高：先生成 patch proposal，等待确认。

### 3. 递归任务分解能力

系统需要把大任务拆成小任务，直到拆成可以直接执行和验证的原子任务。

原子任务必须满足：

- 输入明确。
- 输出明确。
- 前置条件明确。
- 可由一个工具、一个节点、一个服务函数或一次 LLM 判断完成。
- 有可自动判断的完成标准。
- 失败时能返回错误类型和恢复建议。

### 4. 前置条件和依赖判断能力

每个任务执行前必须判断：

- 需要哪些输入。
- 需要哪些文件或数据。
- 需要哪些权限。
- 是否需要用户确认。
- 是否依赖其他子任务结果。
- 是否有冲突条件。
- 是否已有缓存结果可复用。

### 5. 调用栈和返回上级能力

系统要像程序调用栈一样执行任务：

```text
父任务：修改文档
  子任务：找到目标文档
    子任务：列出候选文件
    子任务：按时间和相似度排序
  返回：目标文件路径
  子任务：分析文档结构
  返回：标题层级和段落索引
  子任务：分类新增内容
  返回：建议插入位置
  子任务：执行修改
  返回：patch 和 diff
父任务继续
```

不能因为临时进入某个子任务，就丢失原始目标。

### 6. 结果核对和自我修正能力

每个子任务完成后必须验证：

- 输出是否符合 schema。
- 是否满足完成条件。
- 是否有证据。
- 是否破坏上游约束。
- 是否需要重新规划。

如果验证失败，不应直接结束，而应进入修复策略：

- 重试。
- 降级为更小任务。
- 换工具。
- 请求用户补充。
- 标记 blocked，并保留恢复点。

### 7. 分类能力持续迭代

系统需要维护可版本化的分类规则，例如：

- 文档内容分类：背景、目标、任务、接口、测试、安全、风险、验收。
- 任务类型分类：查找、读取、分析、比对、修改、生成、验证、报告。
- 风险分类：只读、写文件、执行命令、调用外部服务、需要人工确认。
- 失败分类：缺输入、无权限、找不到文件、结果不符合 schema、冲突未解决。

分类规则必须能通过样例和反馈迭代，而不是写死在提示词里。

### 8. 工作区受控写入和 Patch Apply 能力

要让智能体从“能读、能分析、能生成建议”进一步变成“能按授权修改文件”，需要补齐工作区文件写入和 patch apply 层。

该能力必须是受控能力，而不是任意文件系统写入：

- 只能写入白名单工作区根目录。
- 默认先 dry_run，展示将修改哪些文件、哪些行。
- 默认要求 `expected_hash` 或 `expected_version`，防止覆盖别人刚刚改过的文件。
- 默认接入 `ManualApprove` / `HumanReview`，写入和 patch 不允许静默执行。
- 每次写入、patch、失败、回滚都必须写审计记录。

新增能力对象：

- `WorkspaceFileWrite`：按白名单路径写入文件，支持 create / overwrite / append。
- `WorkspacePatchApply`：支持 structured patch 和 unified diff，支持 dry_run / apply / rollback。

这部分能力是 Codex-like 智能体真正具备“修改程序代码、修改文档、生成文件并落盘”的关键补充。

## 三、核心对象模型

### 1. GoalIntent

表示用户原始意图和规范化后的目标。

```json
{
  "goal_id": "goal-001",
  "raw_request": "把最近上次写的某文档找出来，我需要调整...",
  "goal_type": "document_edit",
  "primary_object": "某文档",
  "expected_outcome": "updated_document",
  "constraints": [],
  "risk_level": "medium",
  "requires_user_confirmation": false,
  "confidence": 0.86
}
```

### 2. TaskNode

表示一个可执行任务。

```json
{
  "task_id": "task-001",
  "parent_task_id": "",
  "task_type": "find_file",
  "title": "找到最近一次编写的目标文档",
  "description": "根据文件名线索、修改时间和内容相似度找到候选文档",
  "inputs": {},
  "outputs": {},
  "preconditions": [],
  "completion_criteria": [],
  "status": "pending",
  "risk_level": "low",
  "tool_hint": "WorkspaceFileSearch",
  "evidence": []
}
```

### 3. TaskFrame

表示调用栈中的一帧，用来支持回到上一级任务。

```json
{
  "frame_id": "frame-001",
  "task_id": "task-001",
  "parent_frame_id": "",
  "return_to_task_id": "task-parent",
  "continuation_pointer": "after_find_file",
  "local_context": {},
  "return_value": {},
  "status": "running"
}
```

### 4. Preconditions

前置条件可以是确定性条件，也可以是需要模型判断的条件。

```json
{
  "condition_id": "cond-001",
  "kind": "required_input",
  "field": "target_file_path",
  "required": true,
  "status": "unsatisfied",
  "repair_task_type": "find_file"
}
```

### 5. CompletionCriteria

完成标准必须可验证。

```json
{
  "criteria_id": "crit-001",
  "kind": "schema_match",
  "expected_schema": "TextDocument",
  "status": "pending"
}
```

### 6. VerificationResult

每次执行后输出核对结果。

```json
{
  "task_id": "task-001",
  "passed": true,
  "checks": [],
  "evidence": [],
  "next_action": "return_to_parent"
}
```

## 四、v4 分层架构

### 1. 意图识别与目标规范化层

职责：

- 识别用户目标类型。
- 抽取对象、约束、期望输出。
- 判断是否需要读取文件、修改文档、调用工具、请求确认。
- 给出置信度和不确定项。

建议服务：

- `api/db/services/agent_goal_intent_service.py`
- `agent/component/task_intent.py`

建议节点：

- `GoalIntentClassifier`
- `GoalNormalizer`

### 2. 上下文收集层

职责：

- 根据目标查找相关文件、文档、历史 run、会话消息、模板和已有报告。
- 形成 `TaskContextBundle`。
- 标记证据来源和可信度。

建议服务：

- `api/db/services/agent_task_context_service.py`
- `agent/component/task_context.py`

建议节点：

- `TaskContextCollector`
- `RecentArtifactFinder`
- `RelevantFileResolver`

### 3. 层级任务规划层

职责：

- 把目标拆成任务树或 DAG。
- 判断哪些任务可以并行，哪些必须串行。
- 继续拆分复杂任务，直到原子任务。
- 给每个任务绑定前置条件和完成标准。

建议服务：

- `api/db/services/agent_task_planner_service.py`
- `api/db/services/agent_task_graph_service.py`
- `agent/component/task_planner.py`

建议节点：

- `TaskPlanner`
- `TaskDecomposer`
- `AtomicTaskRefiner`

### 4. 前置条件和依赖层

职责：

- 检查每个任务执行前是否满足条件。
- 自动生成修复子任务。
- 判断权限、文件存在性、输入完整性、上下文充分性。

建议服务：

- `api/db/services/agent_task_precondition_service.py`
- `agent/component/task_preconditions.py`

建议节点：

- `PreconditionChecker`
- `DependencyResolver`

### 5. 执行栈和调度层

职责：

- 维护 task stack。
- 进入子任务。
- 子任务完成后返回父任务。
- 保存 continuation pointer。
- 支持暂停、恢复、重试、跳过、取消。

建议服务：

- `api/db/services/agent_task_stack_service.py`
- `api/db/services/agent_task_execution_service.py`
- `agent/component/task_executor.py`

建议节点：

- `TaskExecutor`
- `TaskStackPush`
- `TaskStackReturn`
- `TaskResume`

### 6. 结果核对和反思层

职责：

- 对每个任务输出做 schema、证据、完成标准核对。
- 判断是否需要重新规划。
- 判断是否需要用户确认。
- 把失败原因结构化。

建议服务：

- `api/db/services/agent_task_verifier_service.py`
- `api/db/services/agent_task_reflection_service.py`
- `agent/component/task_verifier.py`

建议节点：

- `TaskResultVerifier`
- `TaskReflection`
- `ReplanDecider`

### 7. 分类规则和学习层

职责：

- 维护任务类型分类、文档内容分类、风险分类、失败分类。
- 支持版本化。
- 支持人工反馈更新样例。
- 对分类效果做回归评测。

建议服务：

- `api/db/services/agent_task_taxonomy_service.py`
- `api/db/services/agent_task_feedback_service.py`

建议节点：

- `TaskClassifier`
- `ContentClassifier`
- `FeedbackCollector`

### 8. 审计和报告层

职责：

- 记录每次目标识别、分解、执行、核对、重试、返回父任务。
- 输出任务树、执行路径、失败恢复记录。
- 支持下载任务执行报告。

建议服务：

- `api/db/services/agent_task_audit_service.py`
- `api/db/services/agent_task_report_service.py`

建议节点：

- `TaskAuditRecorder`
- `TaskExecutionReportComposer`

### 9. 工作区写入和 Patch 安全层

职责：

- 在白名单根目录内进行文件创建、覆盖、追加和 patch apply。
- 对每次写入进行 dry_run、hash 校验、人工确认、审计记录。
- 支持 structured patch 和 unified diff。
- 支持 patch 失败原因结构化。
- 支持写入前后 diff、备份和回滚。

建议服务：

- `api/db/services/workspace_file_write_service.py`
- `api/db/services/workspace_patch_service.py`
- `api/db/services/workspace_write_audit_service.py`
- 复用 `api/db/services/agent_task_approval_service.py`

建议节点：

- `WorkspaceFileWrite`
- `WorkspacePatchApply`

安全策略：

- 默认 `requires_user_confirmation = true`。
- 默认 `dry_run = true`。
- 默认拒绝写入工作区根目录之外的路径。
- 默认拒绝绝对路径、`..` 路径穿越、符号链接逃逸。
- 默认拒绝 patch 修改超过配置上限的文件数、行数和字节数。

## 六、阶段开发任务

### V4-P0：任务对象模型和持久化底座

目标：

建立 GoalIntent、TaskNode、TaskFrame、Precondition、CompletionCriteria、VerificationResult 的统一数据结构和持久化能力。

任务边界：

- 只做数据模型、状态机、基础 CRUD。
- 不做复杂规划。
- 不接入真实工具执行。

交付范围：

- 新增任务状态枚举：
  - `pending`
  - `ready`
  - `running`
  - `waiting_input`
  - `blocked`
  - `failed`
  - `verified`
  - `completed`
  - `canceled`
- 新增任务关系：
  - parent / child
  - depends_on
  - blocks
  - repair_for
  - return_to
- 新增服务：
  - `AgentTaskModelService`
  - `AgentTaskStateService`
  - `AgentTaskAuditService`

达成标准：

- 可以创建一个 goal。
- 可以创建任务树。
- 可以更新任务状态。
- 可以查询父子关系。
- 可以保存和恢复 task stack。
- 每次状态变化都有 audit 记录。

测试方法：

- 单元测试任务状态流转。
- 单元测试父子任务关系。
- 单元测试非法状态转换会被拒绝。
- 单元测试 task stack push / pop / return。
- API 测试任务创建、查询、状态更新。

建议测试文件：

- `test/unit_test/test_agent_task_model_service.py`
- `test/unit_test/test_agent_task_state_service.py`
- `test/unit_test/test_agent_task_stack_service.py`

### V4-P1：意图识别和目标规范化

目标：

把用户自然语言请求转成结构化 GoalIntent。

任务边界：

- 只识别目标，不执行。
- 只生成目标对象和不确定项。
- 不生成完整任务树。

交付范围：

- 新增 `GoalIntentClassifier`。
- 新增 `GoalNormalizer`。
- 支持目标类型：
  - `find_file`
  - `read_document`
  - `edit_document`
  - `compare_documents`
  - `extract_information`
  - `generate_report`
  - `run_workflow`
  - `ask_question`
  - `needs_clarification`
- 输出字段：
  - goal_type
  - primary_object
  - expected_outcome
  - constraints
  - missing_inputs
  - confidence
  - risk_level
  - requires_user_confirmation

达成标准：

- 对常见任务能稳定输出 goal_type。
- 对不明确任务能输出 missing_inputs，而不是乱执行。
- 对高风险任务能标记 requires_user_confirmation。
- 输出符合固定 JSON schema。

测试方法：

- 构造 50 到 100 条中文任务样例。
- 检查 goal_type 准确率。
- 检查缺失输入识别。
- 检查高风险识别。
- 检查 JSON schema。

建议测试文件：

- `test/unit_test/test_agent_goal_intent_service.py`
- `test/unit_test/test_agent_goal_intent_examples.py`

### V4-P2：上下文收集和目标路径理解

目标：

根据 GoalIntent 自动收集完成任务需要的上下文。

任务边界：

- 可以查找文件、历史 run、已有 artifact、会话上下文。
- 不做任务分解。
- 不修改任何文件。

交付范围：

- 新增 `TaskContextCollector`。
- 新增 `RecentArtifactFinder`。
- 新增 `RelevantFileResolver`。
- 复用 v3 的 WorkspaceFileSearch、WorkspaceFileRead、DocumentNormalizer。
- 输出 `TaskContextBundle`：
  - candidate_files
  - candidate_artifacts
  - recent_runs
  - document_outlines
  - evidence
  - unresolved_context

达成标准：

- 能根据“最近上次写的某文档”找到候选文档。
- 能根据文件名片段、修改时间、内容关键词排序。
- 能输出为什么选择某个候选。
- 找不到时输出 unresolved_context 和建议追问。

测试方法：

- 构造临时文件目录。
- 创建多个候选文件。
- 检查最近修改文件排序。
- 检查内容相似度排序。
- 检查无候选时的恢复建议。
- 检查路径越权仍被拒绝。

建议测试文件：

- `test/unit_test/test_agent_task_context_service.py`
- `test/unit_test/test_agent_relevant_file_resolver.py`

### V4-P3：层级任务分解和原子任务生成

目标：

把 GoalIntent 和 TaskContextBundle 拆成任务树，并继续拆到原子任务。

任务边界：

- 只生成任务计划。
- 不执行任务。
- 不修改文件。

交付范围：

- 新增 `TaskPlanner`。
- 新增 `TaskDecomposer`。
- 新增 `AtomicTaskRefiner`。
- 支持任务树和 DAG。
- 支持串行和并行标记。
- 每个 TaskNode 必须带：
  - task_type
  - inputs
  - outputs
  - preconditions
  - completion_criteria
  - risk_level
  - tool_hint
  - evidence_requirement

原子任务停止条件：

- 单个工具或节点可完成。
- 输入输出明确。
- 完成标准明确。
- 不再包含“分析很多内容”“处理一下”这类模糊动作。

达成标准：

- 对“修改文档并新增内容”能拆成：
  - 找文件
  - 读文件
  - 提取大纲
  - 分类新增内容
  - 生成结构建议
  - 生成修改方案
  - 应用修改或生成 patch proposal
  - 核对 diff
  - 输出报告
- 对复杂任务能递归拆分到原子任务。
- 任务树无孤儿节点。
- 依赖关系无环。

测试方法：

- 任务分解 golden case。
- 检查任务树 schema。
- 检查依赖无环。
- 检查每个叶子任务都有 completion_criteria。
- 检查复杂任务深度不超过配置上限。

建议测试文件：

- `test/unit_test/test_agent_task_planner_service.py`
- `test/unit_test/test_agent_atomic_task_refiner.py`

### V4-P4：前置条件和依赖解析

目标：

执行任务前自动判断条件是否满足，并在不满足时生成修复子任务。

任务边界：

- 只检查条件和生成修复任务。
- 不实际执行修复任务。

交付范围：

- 新增 `PreconditionChecker`。
- 新增 `DependencyResolver`。
- 支持条件类型：
  - required_input
  - file_exists
  - permission_allowed
  - document_loaded

### V4-P17：工作区写入端到端验收和回归测试

目标：

用真实临时工作区验证智能体可以完成“读取文件 -> 生成 patch -> dry_run -> 人工确认 -> apply -> 校验 -> 回滚/审计”的闭环。

任务边界：

- 只在临时工作区运行。
- 不对生产代码目录做真实写入。
- 不执行 shell 命令。

端到端流程：

```text
用户输入：
  找到这个目录里的 agents 页面，把一个入口标题改成智能体创建指南。

系统执行：
  1. GoalIntentClassifier 识别 goal_type = edit_file
  2. WorkspaceFileSearch 找到目标文件
  3. WorkspaceFileRead 读取目标内容
  4. Agent 生成 structured patch
  5. WorkspacePatchApply dry_run
  6. TaskResultVerifier 校验 dry_run 结果
  7. HumanReview 展示变更摘要
  8. ManualApprove 确认
  9. WorkspacePatchApply apply
  10. WorkspaceFileRead 复读文件
  11. TaskResultVerifier 校验变更已落盘
  12. TaskExecutionReportComposer 输出审计报告
```

达成标准：

- 能找到目标文件。
- 能生成 patch plan。
- dry_run 展示受影响文件和变更行。
- 审批前不落盘。
- 审批后落盘。
- hash 冲突时拒绝落盘。
- patch 失败时不产生半写入。
- rollback 能恢复原内容。
- 审计记录可追踪每一步。

测试方法：

- 端到端成功路径。
- 审批拒绝路径。
- hash 不匹配路径。
- patch target 不存在路径。
- 路径越权路径。
- 多文件 patch 部分失败回滚路径。
- rollback 路径。
- 审计完整性路径。

## 八、和 v3 的关系

v3 提供工具能力：

- 文件查找。
- 文件读取。
- 文档规范化。
- 结构化抽取。
- diff。
- 冲突判断。
- 报告和审计。

v4 提供任务智能：

- 知道要做什么。
- 知道先做什么后做什么。
- 知道缺什么条件。
- 知道如何拆到原子任务。
- 知道子任务完成后回到哪里。
- 知道结果是否合格。
- 知道失败后怎么恢复。

v4 本次新增的工作区写入和 patch apply 能力，属于 v4 后段的“受控执行能力”：

- 它依赖 v3 的文件查找、文件读取、文档规范化和 diff 能力。
- 它依赖 v4 前段的任务规划、前置条件、人工确认、执行核对和审计。
- 它不等同于任意命令执行，也不允许绕过白名单直接写系统文件。
- 它是让智能体从“生成修改建议”升级到“在确认后应用修改”的安全执行层。

因此 v4 不替代 v3，而是在 v3 上方增加 Planner / Executor / Verifier / Task Stack 层，并在后段补齐 WorkspaceFileWrite / WorkspacePatchApply 的受控落盘能力。

## 九、首个推荐落地场景

建议第一个真实场景选择：


## 十、工作区写入专项的首个推荐落地场景

建议在 v4 主线稳定后，再启动以下场景：

```text
Codex-like 工作区文件修改智能体
```

用户输入：

```text
找到智能体列表页，把某个六边形入口标题改成“智能体创建指南”，先给我看变更预览，确认后再修改。
```

期望执行路径：

- 识别 goal_type = edit_file。
- 搜索候选文件。
- 读取目标文件。
- 生成 structured patch。
- 执行 dry_run。
- 展示受影响文件、变更行和 diff。
- 等待 ManualApprove。
- 确认后 apply。
- 复读文件校验修改已落盘。
- 输出审计报告和 rollback token。

验收要求：

- 未确认前不得写入。
- hash 不匹配不得写入。
- 路径越权不得写入。
- patch 失败不得半写入。
- 失败时能回到父任务并给出修复建议。
- 成功后能给出 diff、审计记录和回滚方式。
