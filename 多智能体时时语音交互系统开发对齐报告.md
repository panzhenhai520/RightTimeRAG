# 多智能体时时语音交互系统开发对齐报告

生成时间：2026-07-01

核对对象：`多智能体时时语音交互系统详细开发计划.md`

## 1. 总体结论

当前开发进度已经完成 Stage 0 到 Stage 6 的主要工程实现，系统已经从“单 AI 语音链路”推进到具备以下基础能力：

- 稳定的 ASR -> LLM/EKS -> TTS -> playback 基线。
- Kafka / MQTT / Redis / LLM 边界契约。
- turn 状态机、播放仲裁、checkpoint 和 stale drop。
- MeetingState / WorldEvent / meeting event log。
- AI老师 Registry、EKS 路由、voice_profile 路由和 EKS session 隔离。
- 九型人格参数实例化。
- 语言策略选择、生成约束编译和规则质检。

但从完整“真人 + 4 个 AI老师 + 上帝调度 + 教学闭环 + 虚拟数字人”的目标看，当前仍处于地基阶段。Stage 7 之后尚未进入真实闭环开发，上帝调度、真人画像、多 AI老师 proposal、数字人课堂、学习评估、课件制作和课前/课后会议都还没有实现。

## 工程边界声明

本报告中的“已完成修改”和“建议继续推进的服务端任务”，默认均指本机 `voice-project` 里的多AI老师实时语音系统，不默认包含 RAGFlow 前端或 RAGFlow 后端源码修改。

需要特别区分：

- `voice-project 改动`：本机继续实现语音链路、消息事件、AI老师注册、上下文构建、人格策略、上帝调度、真人画像、数字人播放编排、课程闭环和测试脚本。
- `RAGFlow 智能体端改动`：另一台机器上完善 AI 智能体/Chat/Agent/Workflow 的配置、思考流程、节点编排、知识库绑定、输出格式和执行 trace。
- `RAGFlow 后端改动`：另一台机器上修改 RAGFlow 服务端 API、workflow runtime、知识库权限、检索接口、引用返回、执行审计和安全策略。
- `RAGFlow 前端改动`：另一台机器上修改 RAGFlow 画布、节点配置、知识库绑定、权限配置、workflow 调试和 trace 展示界面。

本机代码中的 `workers/ragflow_client.py`、`EKS` 路由和 `ai_teacher_id -> agent/chat/workflow` 映射，只是 `voice-project` 对专家知识库/智能体服务的客户端适配层，不等价于 RAGFlow 后端源码修改。

## 5. 已完成但仍需补强的任务

- worker crash simulation 测试还不完整。

结论：基本达标，但生产级恢复能力仍需补强。

建议后续补强项：

1. 增加 `RecoveryWorker`。
2. 增加 unfinished turn 扫描和状态迁移策略。

## 7. 当前最关键的未达标点

按风险和依赖顺序，当前最关键的未达标点是：

1. Stage 7 未开始。
   - 这是把 Stage 4 AI老师、Stage 5 人格、Stage 6 策略约束真正接入单轮输出的关键。
   - 没有 Stage 7，就不能证明“指定 AI老师能带人格、记忆、策略、约束完成一轮教学发言”。

2. Stage 2 / Stage 3 的恢复能力还不完整。
   - 已有 checkpoint、event log 和 active index。
   - 但缺独立 RecoveryWorker、DB durable event log、crash recovery smoke。

3. Stage 4 还缺真实 4 EKS AI老师端到端验证。
   - 当前是 Registry/Router/session isolation 地基完成。
   - 真实 EKS 四老师、真实 voice_profile、真实知识库绑定还要补。

   - 没有真人画像，上帝调度即使实现，也只能靠规则和即时文本，无法做到因材施教。

   - 当前还没有真正“谁想说、谁该说、谁回答、谁不回答”的决策引擎。

## 8. 推荐下一步顺序

建议严格按以下顺序推进：

1. Stage 7：单AI老师 AITeacherTurnContext 闭环。
   - 归属：`voice-project 改动`为主。
   - 定义 context schema。
   - 注入 AI老师人格摘要、EKS 绑定、voice_profile、语言策略、生成约束、最近对话。
   - 保存 `context_hash` 和 `constraint_hash`。
   - 完成 single-ai_teacher end-to-end smoke。

2. 补强 Stage 4 真实 EKS 四老师验收。
   - 归属：`voice-project`负责注册和路由验收；`RAGFlow 智能体端`负责真实 4 个 AI老师配置。
   - 注册 4 个真实 EKS assistant/chat。
   - 绑定真实 voice_profile。
   - 验证不同老师 EKS session 不串。

3. 补强 Stage 2/3 recovery。
   - 归属：`voice-project 改动`。
   - 实现 RecoveryWorker。
   - 实现 unfinished turn / active meeting recovery smoke。
   - 明确 failed archive。

4. Stage 7A：EKS workflow binding。
   - 归属：`RAGFlow 智能体端/后端/前端改动`为主，`voice-project`只做 workflow 调用适配。
   - 另一台 RAGFlow 机器应先确认 workflow schema 和 API contract，再做画布前端。

5. Stage 8：真人画像和记忆。
   - 归属：`voice-project 改动`为主。

6. Stage 9：AI老师 proposal 与上帝调度。
   - 归属：`voice-project 改动`。

7. Stage 10：多AI老师教学 MVP 闭环。
   - 归属：`voice-project 改动`为主，依赖 `RAGFlow 智能体端`已经提供 4 个可调用 AI老师。

## 9. 可交给另一台 RAGFlow 机器的任务清单

以下任务不应放在本机 `voice-project` 里完成，应该交给正在完善 RAGFlow 智能体功能的机器处理。

### RAGFlow 智能体端改动

1. 创建或完善 4 个真实 AI老师 Agent/Chat/Workflow，并提供稳定的 `agent_id`、`chat_id`、`workflow_id`。
2. 每个 AI老师绑定独立角色、人设、知识库范围、输出格式和可用工具。
3. 支持接收 `AITeacherTurnContext` 中的动态变量：会议 ID、学生输入、最近对话、人格摘要、语言策略、教学目标、用户画像摘要、知识库约束。
4. 支持 AI老师自定义思考流程：LLM 节点、专家知识库检索节点、文档处理节点、代码执行节点、规则判断节点、记忆读写节点、质量检查节点。
5. 支持 workflow 输出统一结构：`answer`、`references`、`trace_summary`、`error_code`、`latency_ms`。
6. 支持课前会议、课堂教学、课后复盘三类 workflow 模板。
7. 不持久化、不暴露 hidden chain-of-thought；只能返回可审计的 `trace_summary`。

### RAGFlow 后端改动

1. 提供稳定的 workflow 调用 API，支持 `session_id`、`context`、`metadata`、`workflow_id`。
2. 实现 workflow schema、runtime、节点执行器、节点超时、失败降级和错误码。
3. 实现老师私有知识库、课程共有知识库、学生记忆空间之间的权限隔离。
4. 返回引用片段、知识库版本、文档来源和检索命中信息。
5. 提供只检索接口，供 `voice-project` 的上帝调度和 gap 分析使用。
6. 增加 workflow 执行审计、权限 guard、prompt injection 防护和 API 鉴权。

### RAGFlow 前端改动

1. 提供 AI老师 workflow 可视化画布，支持节点创建、拖拽、连接和参数配置。
2. 提供节点类型配置界面：LLM、检索、文档处理、代码执行、规则判断、记忆读写、质量检查。
3. 提供每个 AI老师的知识库绑定界面，包括老师私有库和课程共有库。
4. 提供 workflow 调试界面，只展示 `trace_summary`、输入输出摘要、引用和错误码，不展示 hidden chain-of-thought。
5. 提供权限配置界面，管理老师、课程、学生、知识库和 workflow 的访问边界。

## 10. 最终判断

当前系统已经可以作为多AI老师实时语音系统继续开发的工程基础。基础链路、消息边界、状态地基、AI老师注册、人格参数和语言策略均已建立。

但它还不是完整的“真人 + 4 AI老师 + 上帝调度”的教学系统。真正的仿真人多方教学交流，要等 Stage 7、Stage 8、Stage 9、Stage 10 完成后才能成立。

当前最应该继续做的是 Stage 7，而不是直接进入上帝调度或数字人展示。

结论边界：

- 本机 `voice-project` 应独立继续推进 Stage 7、Stage 8、Stage 9、Stage 10 以及 Stage 2/3 recovery 补强。
- RAGFlow 前端/后端/智能体端的改动已经在本报告中单独标注，应交给另一台正在完善 RAGFlow 智能体功能的机器处理。
- 两边通过 API contract、`ai_teacher_id`、`agent_id/chat_id/workflow_id`、`AITeacherTurnContext`、`references` 和 `trace_summary` 对齐，不应把 RAGFlow 源码改动混入本机 `voice-project` 的服务改造任务。
