# 检索增强的意图识别和边界约束算法实现步骤

## 1. 背景与目标

当前聊天助手的 RAG 链路大致是：

```text
用户问题
 -> 多轮问题改写 / 跨语言扩展 / 关键词抽取
 -> _build_retrieval_query()
 -> retriever.retrieval()
 -> _kb_prompt_dynamic()
 -> 大模型生成答案
```

问题是：现有检索主要依赖语义相似度和关键词扩展，容易把“边界相近但约束不满足”的材料召回并交给模型。例如用户问“十八大报告是否有关于信托行业的论述”，系统可能召回“十九大报告”的材料。模型虽然可能说“未找到十八大”，但如果同时引用了召回的 Fig，就会造成证据和结论不一致。

本方案目标是增加一层通用的“约束感知语义解析”能力，不针对某一个问题硬编码，而是泛化处理所有边界约束类问题，包括：

- 时间：2017年底、去年、政策出台后、2012年、近三年。
- 空间：香港、新加坡、亚太、西欧、境内、境外。
- 文档：党的十八大报告、某年度报告、某 PDF、某政策文件。
- 实体：机构、行业、产品、人物、公司、地区组织。
- 版本：新版、旧版、修订版、2023版、现行有效版本。
- 法律条文：法律名称、条号、章节、生效年份、修订年份。
- 会议届数：十八大、十九大、二十大、某届会议。
- 有效范围：合同有效期、政策适用期、条约生效期。

最终要实现：

```text
问题
 -> 问题边界约束解析
 -> 约束归一化
 -> 检索计划生成
 -> 约束参与召回
 -> 约束覆盖校验
 -> 只使用合格证据生成答案
```

## 2. 理论来源与可落地思想

### 2.1 Joint Intent Classification and Slot Filling

来源：BERT for Joint Intent Classification and Slot Filling  
参考：https://arxiv.org/abs/1902.10909

论文思想：

- 意图识别和槽位抽取应联合完成。
- `[CLS]` 表示用于 intent classification。
- token 序列用于 slot filling，常见形式是 BIO 标注。
- 输出不是单一分类，而是一个 semantic frame。

落地到本项目：

- 不只判断 `rag_question`、`comparison`、`temporal_range`。
- 同时抽取问题中的时间、空间、文档、行业、机构、版本、法律条文等槽位。
- 第一阶段可以用“规则 + LLM JSON 抽取”模拟 joint intent-slot 输出；后续积累数据后再训练或微调模型。

### 2.2 TEQUILA Temporal QA

来源：TEQUILA: Temporal Question Answering over Knowledge Bases  
参考：https://arxiv.org/abs/1908.03650

论文思想：

- 时间问题不应直接作为普通语义问题检索。
- 应拆成“普通子问题 + 时间约束”。
- 时间约束需要做区间推理。

落地到本项目：

- “到2017年底，亚太私人财富是否超过西欧”应拆成：
  - 普通子问题：亚太私人财富规模是否超过西欧。
  - 时间约束：截至 2017 年底。
  - 空间约束：亚太、西欧。
- 召回结果必须满足时间和空间边界，不能只凭语义相似。

### 2.3 Faithful Temporal QA

来源：Faithful Temporal Question Answering over Heterogeneous Sources  
参考：https://arxiv.org/abs/2402.15400

论文思想：

- 时间约束不能作为软提示交给模型。
- 对时间问题要强制执行时间条件。
- 答案必须忠实于满足时间条件的证据。

落地到本项目：

- hard constraint 不能只写进 prompt。
- 检索前要增强 query。
- 检索后要检查 chunk 是否满足约束。
- 不满足约束的证据不能进入最终回答 prompt。

### 2.4 TimeML / Temporal Expression Normalization

来源：TimeML、SUTime、HeidelTime 一类时间表达归一化方法  
参考：https://en.wikipedia.org/wiki/TimeML  
参考：https://en.wikipedia.org/wiki/Temporal_expressions

论文和工具思想：

- 把自然语言时间表达规范化成可比较结构。
- 支持日期、区间、持续时间、相对时间、事件时间。

落地到本项目：

- “2017年底”归一为 `2017-12-31` 或 `2017-Q4/end`。
- “去年”需要结合当前日期或问题上下文归一。
- “十八大”归一为会议事件，并派生出会议年份和时间范围。
- “政策出台后”归一为事件相对时间，需要依赖政策发布日期或文档元数据。

### 2.5 GeoSPARQL / Geospatial QA

来源：GeoSPARQL  
参考：https://docs.ogc.org/is/22-047r1/22-047r1.html

思想：

- 地点不是普通关键词，而是结构化空间对象。
- 空间对象之间存在 `within`、`contains`、`intersects` 等关系。

落地到本项目：

- “香港”“新加坡”“亚太”“西欧”需要归一成地点或区域。
- “亚太”可以展开为区域别名和英文表达，但不能随意等同于“全球”。
- 问香港时，新加坡材料最多是背景，不应作为直接证据。

## 4. 数据结构设计

### 4.1 BoundaryFrame

`BoundaryFrame` 是问题语义解析结果。

```json
{
  "intent": "rag_question",
  "answer_type": "existence_check",
  "query_core": "信托行业 发展 政策 论述",
  "slots": {
    "time": [],
    "space": [],
    "document": [],
    "event": [],
    "entity": [],
    "industry": [],
    "organization": [],
    "version": [],
    "law_article": [],
    "effective_range": []
  },
  "confidence": 0.86,
  "requires_knowledge": true
}
```

边界：

- 这里只负责理解问题，不直接改检索结果。
- 不在这一层决定哪些 chunk 可以引用。
- 不把所有名词都塞进 hard constraint，只抽取真正限定问题范围的槽位。

### 4.2 NormalizedConstraint

`NormalizedConstraint` 是槽位归一化后的约束。

```json
{
  "type": "document",
  "surface": "党的十八大报告",
  "canonical": "中国共产党第十八次全国代表大会报告",
  "aliases": ["党的十八大报告", "十八大报告", "第十八次全国代表大会报告"],
  "hard": true,
  "source": "llm+rule",
  "confidence": 0.93
}
```

边界：

- 归一化层允许做别名扩展。
- 不允许把语义相近但届数、年份、版本不同的对象扩展为同义词。
- `hard=true` 的约束必须可解释：来自用户显式表达，或由显式表达确定性派生。

### 4.3 RetrievalPlan

`RetrievalPlan` 是真正进入 RAG 的检索计划。

```json
{
  "primary_query": "信托行业 发展 政策 论述 党的十八大报告 第十八次全国代表大会报告",
  "query_variants": [
    "党的十八大报告 信托行业",
    "第十八次全国代表大会报告 信托",
    "2012 信托行业 政策 论述"
  ],
  "must_groups": [
    ["党的十八大报告", "十八大报告", "第十八次全国代表大会报告"],
    ["信托行业", "信托", "信托公司", "信托业务"]
  ],
  "should_terms": ["政策", "论述", "发展"],
  "near_miss_terms": ["十九大", "第十九次全国代表大会", "党的十九大报告"],
  "metadata_filters": {
    "year": [2012],
    "doc_title_contains_any": ["十八大", "第十八次全国代表大会"]
  },
  "evidence_policy": "direct_evidence_only"
}
```

边界：

- 这是检索层的唯一输入对象。
- `must_groups` 用于约束覆盖检查。
- `metadata_filters` 先支持软接入，后续再改为真正传入 ES / SQL / doc metadata。

### 4.4 EvidenceConstraintResult

每个 chunk 的约束覆盖结果：

```json
{
  "chunk_id": "xxx",
  "constraint_status": "direct_evidence",
  "satisfied_groups": [["十八大报告", "第十八次全国代表大会报告"], ["信托行业", "信托"]],
  "missing_groups": [],
  "near_miss_hits": [],
  "usable_for_answer": true
}
```

状态枚举：

```text
direct_evidence      满足全部 hard constraints
partial_evidence     满足部分 hard constraints
near_miss            命中近似但冲突对象，如十八大问题召回十九大
background_only      可作背景但不能直接证明结论
unrelated            不相关
```

## 阶段 1：实现 BoundaryFrame 解析层

目标：

- 在检索前解析问题的意图和边界槽位。
- 先用“规则 + LLM JSON 抽取”实现，不训练模型。

开发任务：

1. 新建：

```text
api/db/services/question_boundary_service.py
```

2. 实现：

```python
async def parse_question_boundary(
    original_question: str,
    expanded_question: str,
    chat_mdl=None,
) -> dict:
    ...
```

3. 解析策略：

```text
第一层：确定性规则
- 年份、日期、时间范围。
- 明确地点词。
- 常见文档后缀：报告、条例、办法、通知、白皮书、合同、条约。
- 法律条文：第X条、第X章、第X款。
- 版本词：新版、旧版、修订、现行、废止、生效。
- 会议届数：第X届、十八大、十九大、二十大。

第二层：LLM JSON 抽取
- 补充规则没抓到的槽位。
- 判断 answer_type。
- 判断哪些槽位是边界，哪些只是主题词。

第三层：合并去重
- 规则结果优先。
- LLM 结果补充。
- 冲突时保留显式文本，降低置信度。
```

4. 输出 `BoundaryFrame`。

边界：

- 本阶段只解析，不改变检索结果。
- 失败时返回空槽位，不阻断原有 RAG。
- LLM 输出必须做 JSON schema 校验，不能直接信任。

测试：

单元测试输入：

```text
到2017年底，亚太地区的私人财富规模会不会超过西欧？
党的十八大报告有关于信托行业的发展相关的政策或论述吗？
香港吸引家族办公室落户的措施有哪些？
新资本投资者入境计划新版和旧版有什么区别？
《信托法》第十五条在现行版本中如何规定？
某合同在2020年至2023年期间的有效义务是什么？
```

检查输出：

```text
- intent 是否正确。
- answer_type 是否合理。
- time / space / document / event / version / law_article 是否抽取。
- 显式约束是否标记 hard=true。
```

验收条件：

- 测试集中显式时间识别准确率 >= 90%。
- 显式地点识别准确率 >= 90%。
- 文档名/政策名/法律名槽位识别准确率 >= 80%。
- 解析失败不影响原问答链路。

## 阶段 2：实现槽位归一化

目标：

- 把自然语言槽位转换成可检索、可比较、可校验的约束。

开发任务：

1. 实现：

```python
def normalize_boundary_slots(boundary_frame: dict, now=None) -> dict:
    ...
```

2. 时间归一化：

```text
2017年底 -> date=2017-12-31, operator=as_of
2017年 -> range=2017-01-01..2017-12-31
去年 -> 根据当前日期归一为上一自然年
近三年 -> 根据当前日期归一为 rolling_range
政策出台后 -> event_relative，需要后续结合政策发布时间
```

3. 空间归一化：

```text
香港 -> Hong Kong, aliases=[香港, Hong Kong, HK]
新加坡 -> Singapore, aliases=[新加坡, Singapore]
亚太 -> Asia-Pacific, aliases=[亚太, 亚太地区, Asia-Pacific, APAC]
西欧 -> Western Europe, aliases=[西欧, Western Europe]
```

4. 文档归一化：

```text
党的十八大报告 -> canonical document name + aliases
年度报告 -> 尝试结合年份、机构、文档标题
PDF 文件名 -> doc_name_contains
```

5. 法律条文归一化：

```text
《信托法》第十五条 -> law_name=信托法, article=15
2023修订版 -> version_year=2023
现行有效 -> current_effective=true
```

6. 会议届数归一化：

```text
第十八次会议 / 十八大 / 第十八届 -> ordinal=18
派生 aliases、年份或事件名。
```

边界：

- 归一化可以使用小型词典，但词典是通用资源，不是个案补丁。
- 不确定的派生结果标记 `confidence < 0.8`，不作为 hard constraint。
- 不把“相邻年份”“相邻会议届数”“相邻地区”当同义词。

测试：

```text
输入各种时间、地点、文档、法律、版本表达，检查 normalized 结果。
```

验收条件：

- 明确年份、日期、相对年份归一化准确率 >= 95%。
- 常见地点别名归一化准确率 >= 90%。
- 常见文档名别名生成不产生明显错误同义词。
- 不把“十八大”和“十九大”归为同一个 canonical。

## 阶段 3：生成 RetrievalPlan

目标：

- 把 `BoundaryFrame` 和归一化约束转成真实参与检索的计划。

开发任务：

1. 实现：

```python
def build_retrieval_plan(
    original_question: str,
    expanded_question: str,
    normalized_boundary: dict,
) -> dict:
    ...
```

2. 生成字段：

```text
primary_query       主检索 query
query_variants      多路检索 query
must_groups         必须满足的约束别名组
should_terms        提升排序的主题词
near_miss_terms     相邻但不满足约束的词
metadata_filters    可选元数据过滤条件
evidence_policy     direct_evidence_only / allow_background
```

3. 约束强度策略：

```text
用户显式说出的时间、地点、文档、版本、条文 -> hard
由 hard event 确定性派生的年份 -> hard 或 strong
模型猜测出的背景实体 -> soft
普通主题词 -> should
```

4. 近似冲突生成：

```text
时间：相邻年份、不同年份。
会议：十八大 vs 十九大/二十大。
版本：新版 vs 旧版。
地点：香港 vs 新加坡，亚太 vs 西欧。
法律：第十五条 vs 第十六条。
```

注意：near_miss 不是为了排除所有出现这些词的文档，而是为了防止把它们作为直接证据。

边界：

- 不直接依赖某个具体问题。
- 不强制要求所有问题都有 hard constraint；没有边界约束的问题仍走普通 RAG。
- 不把 query 拼得过长，避免破坏向量检索。

测试：

输入 `BoundaryFrame`，检查输出：

```text
- primary_query 包含核心问题和 hard aliases。
- query_variants 能覆盖文档名、时间、地点、主题词组合。
- must_groups 不为空时每组都是可解释边界。
- near_miss_terms 不混入 aliases。
```

验收条件：

- 有显式边界的问题，`RetrievalPlan.must_groups` 不为空。
- 无显式边界的问题，不强行生成错误 hard constraint。
- query 长度受控，不超过现有 `MAX_RETRIEVAL_QUERY_CHARS`。

## 阶段 4：接入检索前增强

目标：

- 让边界约束参与召回，而不是只在回答 prompt 中出现。

开发任务：

1. 在 `dialog_service.py` 中，当前位置：

```python
retrieval_query = _build_retrieval_query(questions[-1], latest_user_question)
```

前面接入：

```python
boundary_frame = await parse_question_boundary(...)
normalized_boundary = normalize_boundary_slots(boundary_frame)
retrieval_plan = build_retrieval_plan(...)
```

2. 用：

```python
retrieval_query = _build_retrieval_query(
    retrieval_plan["primary_query"],
    latest_user_question,
)
```

替换原始 `questions[-1]`。

3. 支持多路检索：

```python
query_variants = retrieval_plan.get("query_variants") or [retrieval_query]
```

先按最小改动实现：

```text
默认只用 primary_query。
如果 hard constraint 存在且初次召回 direct evidence 数量不足，再用 query_variants 补召回。
```

4. 缓存 key 需要加入约束计划摘要：

```text
tenant_id + retrieval_query + kb_ids + boundary_plan_hash
```

否则同一个语义 query 在不同边界下可能复用错误缓存。

边界：

- 第一版不强依赖 ES metadata filter。
- 不改变 retriever.retrieval 的函数签名，先通过 query 增强和后置过滤实现。
- 深度研究模式可先只记录 boundary，不强制应用，后续单独适配。

测试：

```text
对同一主题不同边界提问：
- 香港家族办公室措施
- 新加坡家族办公室措施
- 2017年底私人财富
- 2016年私人财富
- 十八大报告信托行业
- 十九大报告信托行业
```

检查：

```text

- query_variants 是否生成。
- 缓存是否不会串用。
```

验收条件：

- 有 hard constraint 的问题，检索 query 明确带入边界词。
- 不同边界问题不会命中同一个缓存结果。
- 普通无边界问题的 retrieval_query 与原逻辑基本一致。

## 阶段 5：检索后约束覆盖校验

目标：

- 防止向量召回把 near miss 材料交给模型。
- 保证最终 prompt 只包含满足硬约束的证据。

开发任务：

1. 实现：

```python

    ...
```

2. 对每个 chunk 拼接可检查文本：

```text

chunk.content
chunk.docnm_kwd / doc_name
chunk.metadata
```

3. 对 `must_groups` 做覆盖检查：

```python
for group in must_groups:
    satisfied = any(alias in chunk_text for alias in group)
```

4. 分类：

```text
direct_evidence   满足所有 must_groups
partial_evidence  满足部分 must_groups
near_miss         命中 near_miss_terms 且缺少 hard group
background_only   相关但不足以证明
unrelated         不相关
```

5. 过滤策略：

```text
如果 direct_evidence 数量 > 0：
    只把 direct_evidence + 必要背景交给 _kb_prompt_dynamic
如果 direct_evidence 数量 == 0 且 hard constraint 存在：
    不把 near_miss 当证据
    返回空 evidence 或只保留为 audit，不进入 answer prompt
如果没有 hard constraint：
    保持原逻辑
```

6. 在 `_build_evidence_audit` 中加入：

```json

  "constraint_status": "near_miss",
  "missing_constraints": ["党的十八大报告"],
  "satisfied_constraints": ["信托行业"]
}
```

边界：

- 本阶段是必须做的。因为检索前增强不能保证向量召回完全满足约束。
- 不删除原始召回结果，只是不让不合格证据进入生成 prompt。
- 右侧证据分析可以显示 near_miss，但必须标明不是直接证据。

测试：

```text
构造 chunk：

2. 只包含十九大报告和信托行业。
3. 只包含信托行业。
4. 只包含十八大报告。
5. 都不包含。
```

检查分类：

```text
1 -> direct_evidence
2 -> near_miss
3 -> partial_evidence / background_only
4 -> partial_evidence
5 -> unrelated
```

验收条件：

- hard constraint 存在时，不满足 must_groups 的 chunk 不进入最终回答 prompt。
- near_miss 不会被生成答案引用成 Fig。
- direct_evidence 存在时，答案可以正常引用 Fig。
- direct_evidence 不存在时，答案明确说明知识库未找到直接证据。

async def plan_constrained_retrieval(original_question, expanded_question, chat_mdl, now=None):

```python
def build_retrieval_plan(original_question, expanded_question, normalized_boundary):
    core_terms = extract_core_terms(boundary=normalized_boundary, fallback=expanded_question)
    hard_constraints = collect_hard_constraints(normalized_boundary)
    soft_constraints = collect_soft_constraints(normalized_boundary)

    must_groups = []
    for constraint in hard_constraints:
        must_groups.append([constraint["canonical"], *constraint.get("aliases", [])])

    should_terms = dedupe(core_terms + flatten_aliases(soft_constraints))
    near_miss_terms = generate_near_miss_terms(hard_constraints)

    primary_query = compact_query([
        *core_terms,
        *select_top_aliases(hard_constraints),
        *select_top_aliases(soft_constraints),
    ])

    query_variants = build_query_variants(core_terms, hard_constraints)

    return {
        "primary_query": primary_query,
        "query_variants": query_variants,
        "must_groups": must_groups,
        "should_terms": should_terms,
        "near_miss_terms": near_miss_terms,
        "metadata_filters": build_metadata_filters(hard_constraints),
        "evidence_policy": "direct_evidence_only" if must_groups else "standard",
    }
```

```python
def enforce_boundary_constraints(kbinfos, retrieval_plan):
    must_groups = retrieval_plan.get("must_groups") or []
    if not must_groups:
        return kbinfos

    direct = []
    audit_only = []

    for chunk in kbinfos.get("chunks", []):
        result = classify_chunk_by_constraints(chunk, retrieval_plan)
        chunk["constraint_result"] = result

        if result["status"] == "direct_evidence":
            direct.append(chunk)
        else:
            audit_only.append(chunk)

    if direct:
        kbinfos["chunks"] = direct
        kbinfos["constraint_audit_chunks"] = audit_only
        kbinfos["constraint_status"] = "has_direct_evidence"
    else:
        kbinfos["constraint_audit_chunks"] = audit_only
        kbinfos["chunks"] = []
        kbinfos["constraint_status"] = "no_direct_evidence"

    return kbinfos
```

## 7. 关键设计取舍

### 7.1 为什么不能只靠 prompt

因为向量检索会把语义相近材料召回。只在 prompt 中告诉模型“注意十八大”，模型仍然会看到十九大材料，并可能产生混合回答。hard constraint 必须在检索计划和证据过滤中执行。

### 7.2 为什么检索前和检索后都要做

检索前增强可以提高召回到正确证据的概率。  
检索后校验可以防止错误证据进入回答。  
两者缺一不可。

### 7.3 为什么不能所有槽位都 hard

有些实体只是主题词，不是边界。例如“信托行业”既可能是主题，也可能是 hard topic。判断标准：

```text
用户是否用它限定答案范围？
缺少它时答案是否会答非所问？
它是否和时间、空间、文档、版本、条文形成组合约束？
```

### 7.4 没有边界的问题怎么办

没有显式边界的问题继续走普通 RAG。  
但 `BoundaryFrame` 仍可记录：

```json
{
  "time": {"status": "unspecified"},
  "space": {"status": "unspecified"}
}
```

回答时如果召回内容集中在某个范围，可以表述为：

```text
根据当前知识库召回内容，以下主要涉及香港地区……
```

不能把局部知识库材料泛化成全球结论。
