# 智能体自定义平台开发改进计划 v3

生成日期：2026-07-01

## 一、本版调整

本版在前一版多智能体 / AI 教师平台计划基础上，新增一类能力：让 RagFlow 中新建的智能体可以通过配置、流程、节点和函数，实现类似 Codex 的文件读取、文件发现、文档要点抽取、文档差异比对、条款包含关系分析、条款冲突分析和带证据报告输出。

本版只保留开发任务。测试、回归、smoke、端到端验收、压测和用例矩阵已拆分到：

- `多智能体测试任务.md`

## 二、核心结论：FileParser 不能单独完成全部能力

`FileParser` 可以作为“文档内容解析”的一部分，但不能单独解决用户提出的完整需求。

`FileParser` 适合负责：

- 读取已上传或已授权文件的内容。
- 将 PDF / Word / Markdown / txt 等文件拆成文本块。
- 从文件中抽取基础文本、chunk、引用信息。
- 对表格文件做基础读取，或配合 ExcelProcessor 读取表格。

`FileParser` 不适合单独负责：

- 读取服务器指定路径下的任意文件。
- 在指定目录中查找某种类型文件。
- 按文件名、后缀、glob、正则搜索文件。
- 打开某个路径并按行号、段落号、sheet、单元格范围读取。
- 精确比对两份文件的逐行、逐段、逐表格差异。
- 判断 A 条款是否包含 B 条款、B 条款是否被 A 覆盖、两条款是否冲突。
- 将法律条款和合同条款做证据可追溯的语义冲突分析。
- 管理文件系统权限、路径白名单、审计记录。

因此需要新增一组服务和节点，而不是只扩展一个 FileParser。

## 三、目标能力

目标是让 RagFlow 智能体可以在低代码流程中配置以下能力：

- 指定一个安全根目录或知识库目录。
- 查找指定路径下的文件。
- 按后缀、文件名、正则、修改时间、大小等条件筛选文件。
- 打开文件并读取内容。
- 对文本按行、段落、标题层级、页码读取。
- 对表格读取为结构化数据，包括 sheet、表头、行、列、单元格范围。
- 对 PDF / Word / Markdown / txt / Excel 做统一规范化。
- 抽取观点、条款、定义、义务、权利、禁止、条件、期限、金额等结构化信息。
- 对两份文档做精确 diff。
- 对两份文档做语义匹配。
- 判断条款关系：相同、等价、A 包含 B、B 包含 A、缺失、额外、范围更宽、范围更窄、冲突、无法判断。
- 生成带原文证据位置的报告。
- 将所有文件访问、比对结果和报告产物纳入审计。

## 四、分层架构

### 1. 文件工作区访问层

新增 Workspace/File 能力，解决“指定路径、查找文件、打开文件、按范围读取”的问题。

建议服务：

- `api/db/services/workspace_file_service.py`
- `api/apps/restful_apis/workspace_file_api.py`
- `agent/component/workspace_file.py`

建议 API：

```text
GET  /workspace/roots
POST /workspace/files/list
POST /workspace/files/search
POST /workspace/files/stat
POST /workspace/files/read
POST /workspace/files/read-range
POST /workspace/files/read-table
POST /workspace/files/compare-inputs
```

安全要求：

- 只能访问配置白名单中的根目录。
- 禁止路径穿越，例如 `../`、软链逃逸、绝对路径越权。
- 每次文件访问记录 tenant、user、agent、run、path、action、allow/deny、reason。
- 大文件读取必须分页、截断或分块。
- 表格读取必须限制最大 sheet、最大行列、最大单元格数。

### 2. 文档读取与规范化层

这一层复用并扩展 `FileParser`，但输出必须比当前 chunk 更稳定，方便后续精确定位和比对。

建议服务：

- `api/db/services/document_normalize_service.py`
- `agent/component/document_normalizer.py`

统一输出结构：

```json
{
  "document_id": "doc-a",
  "filename": "contract-a.docx",
  "mime_type": "application/vnd.openxmlformats-officedocument.wordprocessingml.document",
  "source_path": "/safe/root/contracts/contract-a.docx",
  "pages": [],
  "sections": [],
  "paragraphs": [],
  "lines": [],
  "tables": [],
  "chunks": [],
  "metadata": {}
}
```

文本块需要包含：

```json
{
  "block_id": "doc-a:p12",
  "block_type": "paragraph",
  "text": "原文内容",
  "normalized_text": "规范化内容",
  "page": 3,
  "line_start": 120,
  "line_end": 126,
  "paragraph_index": 12,
  "section_path": ["第二章", "付款条款"],
  "char_start": 3580,
  "char_end": 3820,
  "source_ref": "contract-a.docx | page 3 | paragraph 12"
}
```

表格块需要包含：

```json
{
  "table_id": "doc-a:t1",
  "sheet_name": "报价表",
  "headers": ["项目", "数量", "单价", "总价"],
  "rows": [
    {"row_index": 2, "values": {"项目": "服务费", "数量": 1, "单价": 10000, "总价": 10000}}
  ],
  "source_ref": "quote.xlsx | sheet 报价表 | rows 2-20"
}
```

### 3. 结构化抽取层

这一层负责从规范化后的文档中抽取观点、条款和可比对要素。

建议组件：

- `ViewpointExtractor`
- `ClauseExtractor`
- `DefinitionExtractor`
- `ObligationExtractor`
- `RiskPointExtractor`
- `TableFactExtractor`

建议输出：

```json
{
  "item_id": "A-12",
  "document_id": "doc-a",
  "item_type": "obligation",
  "text": "乙方应在收到发票后三十日内付款。",
  "normalized_text": "乙方在收到发票后30日内付款",
  "subject": "乙方",
  "action": "付款",
  "object": "合同款项",
  "condition": "收到发票后",
  "deadline": "30日内",
  "amount": null,
  "legal_effect": "付款义务",
  "evidence": {
    "source_ref": "contract-a.docx | page 3 | paragraph 12",
    "page": 3,
    "paragraph_index": 12,
    "line_start": 120,
    "line_end": 126
  },
  "confidence": 0.91
}
```

抽取节点可以使用 LLM，但必须输出固定 JSON schema，并保留原文证据位置。

### 4. 比对与冲突判断层

这一层分为确定性比对和语义比对。

确定性比对负责：

- 文件 hash 比对。
- 行级 diff。
- 段落级 diff。
- 表格行列 diff。
- 标题结构 diff。
- 文档 A 有、文档 B 无的块。
- 文档 B 有、文档 A 无的块。

语义比对负责：

- 条款匹配。
- 观点匹配。
- 包含与被包含关系。
- 范围更宽 / 更窄。
- 条款冲突。
- 合同条款是否违反法律条款。
- 合同条款是否遗漏法律强制要求。

建议组件：

- `DocumentDiff`
- `ParagraphDiff`
- `TableDiff`
- `ClauseMatcher`
- `SemanticComparer`
- `ConflictDetector`
- `LegalContractConflictDetector`

建议关系枚举：

```text
same
equivalent
a_contains_b
b_contains_a
a_broader_than_b
a_narrower_than_b
missing_in_a
missing_in_b
extra_in_a
extra_in_b
conflict
ambiguous
unmatched
```

建议输出：

```json
{
  "a_item_id": "LAW-15",
  "b_item_id": "CONTRACT-08",
  "relation": "conflict",
  "severity": "high",
  "summary": "合同约定付款期限为90日，法律条款要求不得超过30日。",
  "reason": "B 条款的履行期限长于 A 条款强制上限。",
  "a_evidence": {
    "source_ref": "law.pdf | page 5 | article 15"
  },
  "b_evidence": {
    "source_ref": "contract.docx | page 3 | paragraph 8"
  },
  "confidence": 0.88
}
```

### 5. 编排、报告和审计层

Agent workflow 可以把以上节点编排起来。

示例流程：

```text
Begin
  -> WorkspaceFileSearch
  -> WorkspaceFileRead
  -> DocumentNormalizer(A)
  -> DocumentNormalizer(B)
  -> ClauseExtractor(A)
  -> ClauseExtractor(B)
  -> DocumentDiff
  -> ClauseMatcher
  -> ConflictDetector
  -> CitationFormatter
  -> ReportComposer
  -> DocGenerator
```

报告至少包含：

- 文件清单。
- 解析状态。
- 结构化抽取结果。
- 精确 diff 摘要。
- 条款匹配矩阵。
- 包含与被包含关系。
- 冲突清单。
- 缺失条款。
- 高风险条款。
- 每个结论的原文证据位置。
- 下载链接。

审计至少记录：

- 谁访问了哪个文件。
- 哪个 Agent / run 做了哪次比对。
- 使用了哪些模型和节点。
- 生成了哪些报告产物。
- 哪些结论来自确定性算法，哪些结论来自 LLM 判断。

## 五、v3 开发任务

### V3-P0：落地 4 个真实默认 AI 老师

目标：从“默认教师注册配置”升级为“真实可调用的 4 个 Agent / Workflow / Session”。

当前缺口：

- `AgentTeacherRegistryService.DEFAULT_TEACHERS` 已定义 4 个稳定教师 ID。
- 这些记录仍偏配置契约，缺少可确认的真实 workflow canvas、release version 和默认会话创建策略。

交付范围：

- 提供幂等初始化脚本或管理 API，将 4 个教师实例化：
  - `ai_teacher_lead`
  - `ai_teacher_phonetics`
  - `ai_teacher_grammar`
  - `ai_teacher_coach`
- 每个教师绑定稳定：
  - `agent_id`
  - `workflow_id`
  - release version
  - persona / role / language_style / default_intentions
  - dataset role 配置
- `/agents/teachers/defaults` 返回每个教师是否已实例化、是否可访问、当前 release version、默认 workflow id。
- 初始化流程必须幂等，重复执行不会创建重复 Agent。

完成条件：

- 4 个教师都有稳定可调用 ID。
- 缺失或未发布的教师能在 defaults API 中明确暴露状态。

### V3-P1：AI 教师结构化输出强校验

目标：把“模板提示模型输出 JSON”升级为“服务端强制契约”。

交付范围：

- 新增或扩展 AI 教师输出校验服务：
  - required：`answer`、`intention`、`target`、`confidence`
  - intention 必须属于教师 registry enum
  - confidence 限制 0 到 1
  - `reply_to` 只允许空、`student`、`all`、`god`、`teacher:<id>`
  - `target` / `target_listener` 必须和上下文一致
  - 禁止隐藏 CoT、系统提示、敏感 token 出现在 public 输出中
- 校验失败策略：
  - 可修复字段自动规范化并记录 warning。
  - 关键字段缺失或非法时返回标准错误码。
- 校验结果写入 run trace，public trace 只暴露 hash、错误码和字段名。

完成条件：

- public response 中的教师输出字段稳定。
- 非法字段有确定性的处理策略。

### V3-P2：知识库权限隔离和 dataset role 映射

目标：把“检索节点支持 request scope”升级为“AI 教师知识库隔离可审计”。

交付范围：

- public invoke 和 meeting runs 入口校验 `request_dataset_ids`。
- 调用方无权访问的 dataset 直接拒绝。
- 请求 dataset 与节点配置无交集时返回空检索，不回退到节点默认 dataset。
- 建立 dataset role 映射：
  - `course_shared`
  - `lesson_plan`
  - `teacher_private`
  - `textbook`
  - `student_record`
- 每个教师只可访问自己 role 允许的 dataset。
- 检索引用统一包含：
  - `dataset_id`
  - `document_id`
  - `chunk_id`
  - `source_ref`
  - `score`
  - `page`
  - `version`
  - `article_no`
  - `metadata_incomplete`
- 记录 dataset access audit。

完成条件：

- 教师私有知识库不会被其他教师读取。
- public response references 字段稳定且可追溯。

### V3-P3：voice-project 会议 fan-in 统一结果合约

目标：让 voice-project 不需要理解 RagFlow 内部 trace，就能消费多教师会议结果。

交付范围：

- 扩展 `/agents/meetings/results` 输出：
  - `meeting_id`
  - `turn_id`
  - `status`
  - `reply_text`
  - `memory_delta`
  - `citations`
  - `score_result`
  - `voice` 或 `voice_params`
  - `agent_results`
  - `failed_agents`
  - `trace_summary`
- 每个 `agent_results` 使用 `AgentPublicResponseService` 的稳定 response shape。
- 支持部分失败：
  - 一个教师超时或失败，不阻塞其他教师结果。
  - 总响应包含 `partial_failed` 状态和可重试错误码。
- 支持按 teacher role 排序和聚合策略。

完成条件：

- voice-project 调一次 results API 就能拿到可播报文本、引用、记忆增量和失败信息。
- `agent_results` 不暴露内部 inputs / outputs / hidden thoughts。

### V3-P4：长期记忆和写入确认策略

完成条件：

- 长期记忆有来源、确认人、run id 和删除能力。
